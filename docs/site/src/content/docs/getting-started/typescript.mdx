ESM-only · Node 20+ · ships as a native addon (per-platform binary).

`zod` is a peer dependency for the typed-validation facade (you bring your own Zod version). You supply
a Zod schema at render time; the library imports only the structural `safeParse` interface, not Zod's
identity, so any object exposing `safeParse` works.

## Write a prompt definition

A prompt definition is YAML, JSON, or TOML text — same shape, your choice of format. You read the text
however suits your app and hand it to the `from*` factories; the library does no file I/O itself. The
`greet` definition used throughout this page:

<Tabs syncKey="def-format">
<TabItem label="YAML">
```yaml title="greet.yaml"
name: greet
role: user
body: "Hi {{ name }}, you have {{ count }} messages."
variables:
  name:
    type: string
    origin: trusted
  count:
    type: integer
    origin: trusted
```
</TabItem>
<TabItem label="JSON">
```json title="greet.json"
{
  "name": "greet",
  "role": "user",
  "body": "Hi {{ name }}, you have {{ count }} messages.",
  "variables": {
    "name": { "type": "string", "origin": "trusted" },
    "count": { "type": "integer", "origin": "trusted" }
  }
}
```
</TabItem>
<TabItem label="TOML">
```toml title="greet.toml"
name = "greet"
role = "user"
body = "Hi {{ name }}, you have {{ count }} messages."

[variables.name]
type = "string"
origin = "trusted"

[variables.count]
type = "integer"
origin = "trusted"
```
</TabItem>
</Tabs>

## Construct and render

### Step 1 — construct a `Prompt` (and validate)

Construct from text with a `from*` factory, or from a typed `PromptDefinition` object with
`new Prompt(...)`. Both run the **same** validation immediately:

<Tabs syncKey="construct">
<TabItem label="From text">
```ts
import { Prompt } from "prompting-press";
import { readFileSync } from "node:fs";

// You read the text however suits your app; the library does no file I/O itself.
const greet = Prompt.fromYaml(readFileSync("greet.yaml", "utf-8")); // validates here, or throws
// fromJson / fromToml parse the JSON / TOML forms into the same Prompt.
// (TOML is parsed by Rust — no JS TOML dependency.)

greet.name; // => "greet"
greet.body; // => "Hi {{ name }}, you have {{ count }} messages."
```
</TabItem>
<TabItem label="From an object">
```ts
import { Prompt, type PromptDefinition } from "prompting-press";

// The constructor takes a typed PromptDefinition — your editor type-checks the
// shape (field names, the role enum, each variable's `origin`) as you write it.
const definition: PromptDefinition = {
  name: "greet",
  role: "user",
  body: "Hi {{ name }}, you have {{ count }} messages.",
  variables: {
    name: { type: "string", origin: "trusted" },
    count: { type: "integer", origin: "trusted" },
  },
};

const greet = new Prompt(definition); // same validation as the from* factories

greet.name; // => "greet"
```
</TabItem>
</Tabs>

Construction **validates immediately**: had the body referenced `{{ usrname }}` — a variable
`variables` doesn't declare — construction would have **thrown** a `PromptRenderError`, **before any
render** (malformed text throws `LoadError`). A template/variable disagreement is caught at
construction, never as a silent empty render.

### Step 2 — declare the typed Vars schema

The values you render with are a Zod schema you own. Its keys match the prompt's `variables`
(`name`, `count`), and `safeParse` runs before the kernel is touched:

```ts
import { z } from "zod";

const GreetVars = z.object({
  name: z.string().min(1),
  count: z.number().int().nonnegative(),
});
```

### Step 3 — render, and read the result

Hand the `Prompt` from Step 1 the `GreetVars` schema from Step 2 plus the data — Zod validates, then
the kernel renders:

```ts
const result = greet.render(GreetVars, { name: "Ada", count: 3 });

result.text;         // => "Hi Ada, you have 3 messages."
result.variant;      // => "default"  (same arm greet.body showed in Step 1)
result.templateHash; // => e.g. "9f2c…"  (64-char lowercase-hex SHA-256 of the template)
result.renderHash;   // => e.g. "1ab4…"  (64-char lowercase-hex SHA-256 of result.text)
result.guard;        // => null  (no guard requested)
```

`render` has two forms — `variant`/`guard` ride in an optional `opts` object:

```ts
prompt.render(schema, data, opts?);  // schema form: Zod validates data, then renders
prompt.render(data, opts?);          // static form: already-typed data, marshaled directly
// opts: { variant?: string; guard?: GuardConfig | null }
```

`variant` absent selects the default (root body) arm; `guard` absent opts out of the advisory guard.

<Aside type="note">
**TS field naming:** `RenderResult` uses camelCase (`templateHash`, `renderHash`) because napi-rs
maps Rust `snake_case` to JavaScript conventions. The Rust and Python bindings use `template_hash` /
`render_hash`.
</Aside>

### Complete example

```ts
import { Prompt } from "prompting-press";
import { z } from "zod";
import { readFileSync } from "node:fs";

const GreetVars = z.object({
  name: z.string().min(1),
  count: z.number().int().nonnegative(),
});

// 1. Construct (validates here).
const greet = Prompt.fromYaml(readFileSync("greet.yaml", "utf-8"));

// 2 + 3. Render with the typed, Zod-validated vars.
const result = greet.render(GreetVars, { name: "Ada", count: 3 });
console.log(result.text);         // Hi Ada, you have 3 messages.
console.log(result.templateHash); // 64-char hex
```

## What `RenderResult` carries

| Field | Value |
|-------|-------|
| `text` | The rendered body text. |
| `variant` | The resolved variant name (`"default"` when none was selected). |
| `templateHash` | `SHA256(resolved variant template source)` — lowercase hex. |
| `renderHash` | `SHA256(rendered output text)` — lowercase hex. |
| `guard` | `string | null` — advisory guard text when opted in; `null` otherwise. |

## Error types

```ts
import {
  Prompt,
  PromptingPressError,
  PromptValidationError,
  PromptRenderError,
  LoadError,
} from "prompting-press";
import { z } from "zod";

try {
  greet.render(GreetVars, { name: "Ada", count: -1 });
} catch (err) {
  if (err instanceof PromptValidationError) {
    for (const row of err.errors) {
      console.error(row.field, row.code, row.message);
      // "count"  "validation"  "Number must be greater than or equal to 0"
    }
  }
}
```

The error hierarchy (all extend `Error`):

```
PromptingPressError          # base; carries .errors: FieldError[]
├── PromptValidationError    # Zod validation failed      (code "validation")
├── PromptRenderError        # kernel render failure       (codes: unknown_variant,
│                            #   undefined_variable, parse, render, excluded_feature)
├── UnknownPromptError       # a prompt name lookup failed (code "unknown_prompt")
└── LoadError                # malformed YAML/JSON/TOML   (code "load")
```

Each `FieldError` carries `field`, `code`, and `message`. Sensitive rejected values are
**never** echoed onto the error surface.
