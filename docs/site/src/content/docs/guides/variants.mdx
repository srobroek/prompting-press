A **variant** is a named alternative *body* for a prompt that shares the same declared `variables`,
`role`, and `output_model`. Variants are the mechanism for parallel, coexisting wordings of the same
prompt (`concise`, `formal`, an experiment arm) — the caller selects one by name at render time.

This guide covers **declaring** variants in a prompt document and **selecting** one at render. To add
a variant to an already-constructed `Prompt` at runtime, see
**[Deriving a prompt](/guides/derive/)**.

## The default arm

Every prompt has an implicit default arm: the **root `body`**. It is surfaced under the reserved
variant name `"default"`, and it renders when no variant is selected
(`variant = None` / omitted). A variant literally named `default` is rejected at construction — that
name is reserved for the root body.

## Declaring variants in the document

Add a `variants` map to the prompt definition. Each entry has its **own `body`** and may carry an
opaque per-variant `metadata` map. Variants share the prompt's `variables`, `role`, and
`output_model` — a variant that tries to redefine those is rejected (it differs **only** in
body + metadata).

<Tabs syncKey="def-format">
<TabItem label="YAML">
```yaml title="summary.yaml"
name: summary
role: user
body: "Summarise the following article in {{ max_words }} words:\n\n{{ article }}"
variables:
  article:
    type: string
    trusted: false
  max_words:
    type: integer
    trusted: true
variants:
  concise:
    body: "In one sentence, summarise: {{ article }}"
  structured:
    body: "Summarise {{ article }} as a title, three bullets, and a one-line conclusion."
    metadata:
      group: experiment-q4   # opaque to the library — the caller interprets it
```
</TabItem>
<TabItem label="JSON">
```json title="summary.json"
{
  "name": "summary",
  "role": "user",
  "body": "Summarise the following article in {{ max_words }} words:\n\n{{ article }}",
  "variables": {
    "article": { "type": "string", "trusted": false },
    "max_words": { "type": "integer", "trusted": true }
  },
  "variants": {
    "concise": { "body": "In one sentence, summarise: {{ article }}" },
    "structured": {
      "body": "Summarise {{ article }} as a title, three bullets, and a one-line conclusion.",
      "metadata": { "group": "experiment-q4" }
    }
  }
}
```
</TabItem>
<TabItem label="TOML">
```toml title="summary.toml"
name = "summary"
role = "user"
body = "Summarise the following article in {{ max_words }} words:\n\n{{ article }}"

[variables.article]
type = "string"
trusted = false

[variables.max_words]
type = "integer"
trusted = true

[variants.concise]
body = "In one sentence, summarise: {{ article }}"

[variants.structured]
body = "Summarise {{ article }} as a title, three bullets, and a one-line conclusion."
metadata = { group = "experiment-q4" }   # opaque to the library — the caller interprets it
```
</TabItem>
</Tabs>

This prompt has three arms: the default (the root `body`), `concise`, and `structured`. All three see
the same `article` / `max_words` variables — the agreement check covers every arm, and a variant
body that referenced an undeclared variable would fail at construction.

## Constructing inline from a shape object

The same three-arm prompt can be built directly from a typed shape object — no text to parse. The
`variants` map mirrors the document form one-to-one, and construction runs the **same** validation
(agreement over every arm, reserved-name check).

<Tabs syncKey="construct">
<TabItem label="Rust">
```rust
use prompting_press::{Prompt, PromptDefinition};

let def: PromptDefinition = serde_json::from_value(serde_json::json!({
    "name": "summary",
    "role": "user",
    "body": "Summarise the following article in {{ max_words }} words:\n\n{{ article }}",
    "variables": {
        "article":   { "type": "string",  "trusted": false },
        "max_words": { "type": "integer", "trusted": true }
    },
    "variants": {
        "concise":    { "body": "In one sentence, summarise: {{ article }}" },
        "structured": {
            "body": "Summarise {{ article }} as a title, three bullets, and a one-line conclusion.",
            "metadata": { "group": "experiment-q4" }
        }
    }
}))?;

let summary = Prompt::new(def)?;   // same validation as Prompt::from_yaml
summary.variants().contains_key("concise");   // true
```
</TabItem>
<TabItem label="Python">
```python
from prompting_press import Prompt

summary = Prompt({
    "name": "summary",
    "role": "user",
    "body": "Summarise the following article in {{ max_words }} words:\n\n{{ article }}",
    "variables": {
        "article":   {"type": "string",  "trusted": False},
        "max_words": {"type": "integer", "trusted": True},
    },
    "variants": {
        "concise":    {"body": "In one sentence, summarise: {{ article }}"},
        "structured": {
            "body": "Summarise {{ article }} as a title, three bullets, and a one-line conclusion.",
            "metadata": {"group": "experiment-q4"},
        },
    },
})   # same validation as Prompt.from_yaml
list(summary.variants)   # ["concise", "structured"]
```
</TabItem>
<TabItem label="TypeScript">
```ts
import { Prompt, type PromptDefinition } from "prompting-press";

const definition: PromptDefinition = {
  name: "summary",
  role: "user",
  body: "Summarise the following article in {{ max_words }} words:\n\n{{ article }}",
  variables: {
    article: { type: "string", trusted: false },
    max_words: { type: "integer", trusted: true },
  },
  variants: {
    concise: { body: "In one sentence, summarise: {{ article }}" },
    structured: {
      body: "Summarise {{ article }} as a title, three bullets, and a one-line conclusion.",
      metadata: { group: "experiment-q4" },
    },
  },
};

const summary = new Prompt(definition);   // same validation as Prompt.fromYaml
Object.keys(summary.variants);            // ["concise", "structured"]
```
</TabItem>
</Tabs>

## Selecting a variant at render

Pass the variant name at render; omit it (or pass `None` / no `variant`) for the default arm. The
resolved name comes back on `RenderResult.variant`.

<Tabs>
<TabItem label="Rust">
```rust
let summary = Prompt::from_yaml(yaml_text)?;

let def = summary.render(&vars, None, &GuardConfig::default())?;          // default (root body)
let concise = summary.render(&vars, Some("concise"), &GuardConfig::default())?;

def.variant;     // => "default"
concise.variant; // => "concise"
```
An unknown variant name returns `Err(ConsumerError::Kernel(..))` with code `"unknown_variant"`.
</TabItem>
<TabItem label="Python">
```python
summary = Prompt.from_yaml(yaml_text)

default = summary.render(Vars, data=data)                    # default (root body)
concise = summary.render(Vars, data=data, variant="concise")

default.variant   # => "default"
concise.variant   # => "concise"
```
An unknown variant name raises `PromptRenderError` with code `"unknown_variant"`.
</TabItem>
<TabItem label="TypeScript">
```ts
const summary = Prompt.fromYaml(yamlText);

const def = summary.render(Vars, data);                       // default (root body)
const concise = summary.render(Vars, data, { variant: "concise" });

def.variant;     // => "default"
concise.variant; // => "concise"
```
An unknown variant name throws `PromptRenderError` with code `"unknown_variant"`.
</TabItem>
</Tabs>

The variant name is a plain string resolved at render time — the set of variants is whatever the
prompt document declares, so it is not a compile-time enum. The names are readable from the prompt
object, and an unknown name fails loudly, never silently.

## Discovering the available variants

The `variants` accessor returns the declared variant map; read its keys to see what is selectable
(the default arm is always available and is not listed there — it is the root body, name `"default"`).

<Tabs>
<TabItem label="Rust">
```rust
summary.variants().keys().collect::<Vec<_>>();   // ["concise", "structured"]
summary.variants().contains_key("concise");      // true
```
</TabItem>
<TabItem label="Python">
```python
list(summary.variants)              # ["concise", "structured"]
"concise" in summary.variants       # True
```
</TabItem>
<TabItem label="TypeScript">
```ts
Object.keys(summary.variants);      // ["concise", "structured"]
"concise" in summary.variants;      // true
```
</TabItem>
</Tabs>

Passing a name that is not declared returns/raises/throws a structured error with code
`"unknown_variant"` (shown per language above) — bad selections surface immediately.

<Aside type="note" title="Selection is caller-owned">
The library validates the variant name and renders that arm — it does **not** own
experiment-assignment logic or select a variant automatically. Which arm to render (an A/B split, a
feature flag, a per-tenant choice) is the application's decision; pass the chosen name.
</Aside>

## Per-variant metadata

Each variant can carry an opaque `metadata` map (weights, group labels) for the caller's selection
logic — the library stores and echoes it but never acts on it. See
**[Metadata](/guides/metadata/)** for how per-variant and prompt-level `metadata`
work and how to read them back.
