A **variant** is a named alternative *body* for a prompt that shares the same declared `variables`,
`role`, and `output_model`. Variants are the one multi-template need git cannot express well — a set
of parallel, coexisting wordings of the same prompt (`concise`, `formal`, an experiment arm) that you
choose between at render time.

This guide covers **declaring** variants in a prompt document and **selecting** one at render. To add
a variant to an already-constructed `Prompt` at runtime, see
**[Deriving a prompt](/prompting-press/guides/derive/)**.

## The default arm

Every prompt has an implicit default arm: the **root `body`**. It is surfaced under the reserved
variant name `"default"`, and it is what renders when you don't select a variant
(`variant = None` / omitted). You never declare a variant literally named `default` — that name is
reserved for the root body, and a prompt that tries to is rejected at construction.

## Declaring variants in the document

Add a `variants` map to the prompt definition. Each entry has its **own `body`** and may carry opaque
per-variant `meta`. Variants share the prompt's `variables`, `role`, and `output_model` — a variant
that tries to redefine those is rejected (it differs **only** in body + meta).

<Tabs syncKey="def-format">
<TabItem label="YAML">
```yaml title="summary.yaml"
name: summary
role: user
body: "Summarise the following article in {{ max_words }} words:\n\n{{ article }}"
variables:
  article:
    type: string
    origin: untrusted
  max_words:
    type: integer
    origin: trusted
variants:
  concise:
    body: "In one sentence, summarise: {{ article }}"
  structured:
    body: "Summarise {{ article }} as a title, three bullets, and a one-line conclusion."
    meta:
      group: experiment-q4   # opaque to the library — your code interprets it
```
</TabItem>
<TabItem label="JSON">
```json title="summary.json"
{
  "name": "summary",
  "role": "user",
  "body": "Summarise the following article in {{ max_words }} words:\n\n{{ article }}",
  "variables": {
    "article": { "type": "string", "origin": "untrusted" },
    "max_words": { "type": "integer", "origin": "trusted" }
  },
  "variants": {
    "concise": { "body": "In one sentence, summarise: {{ article }}" },
    "structured": {
      "body": "Summarise {{ article }} as a title, three bullets, and a one-line conclusion.",
      "meta": { "group": "experiment-q4" }
    }
  }
}
```
</TabItem>
<TabItem label="TOML">
```toml title="summary.toml"
name = "summary"
role = "user"
body = "Summarise the following article in {{ max_words }} words:\n\n{{ article }}"

[variables.article]
type = "string"
origin = "untrusted"

[variables.max_words]
type = "integer"
origin = "trusted"

[variants.concise]
body = "In one sentence, summarise: {{ article }}"

[variants.structured]
body = "Summarise {{ article }} as a title, three bullets, and a one-line conclusion."
meta = { group = "experiment-q4" }   # opaque to the library — your code interprets it
```
</TabItem>
</Tabs>

This prompt has three arms: the default (the root `body`), `concise`, and `structured`. All three see
the same `article` / `max_words` variables — so the agreement check covers every arm, and a variant
body that referenced an undeclared variable would fail at construction.

## Or construct inline from the shape object

The same three-arm prompt can be built directly from a typed shape object — no text to parse. The
`variants` map mirrors the document form one-to-one, and construction runs the **same** validation
(agreement over every arm, reserved-name check).

<Tabs syncKey="construct">
<TabItem label="Rust">
```rust
use prompting_press::{Prompt, PromptDefinition};

let def: PromptDefinition = serde_json::from_value(serde_json::json!({
    "name": "summary",
    "role": "user",
    "body": "Summarise the following article in {{ max_words }} words:\n\n{{ article }}",
    "variables": {
        "article":   { "type": "string",  "origin": "untrusted" },
        "max_words": { "type": "integer", "origin": "trusted" }
    },
    "variants": {
        "concise":    { "body": "In one sentence, summarise: {{ article }}" },
        "structured": {
            "body": "Summarise {{ article }} as a title, three bullets, and a one-line conclusion.",
            "meta": { "group": "experiment-q4" }
        }
    }
}))?;

let summary = Prompt::new(def)?;   // same validation as Prompt::from_yaml
summary.variants().contains_key("concise");   // true
```
</TabItem>
<TabItem label="Python">
```python
from prompting_press import Prompt

summary = Prompt({
    "name": "summary",
    "role": "user",
    "body": "Summarise the following article in {{ max_words }} words:\n\n{{ article }}",
    "variables": {
        "article":   {"type": "string",  "origin": "untrusted"},
        "max_words": {"type": "integer", "origin": "trusted"},
    },
    "variants": {
        "concise":    {"body": "In one sentence, summarise: {{ article }}"},
        "structured": {
            "body": "Summarise {{ article }} as a title, three bullets, and a one-line conclusion.",
            "meta": {"group": "experiment-q4"},
        },
    },
})   # same validation as Prompt.from_yaml
list(summary.variants)   # ["concise", "structured"]
```
</TabItem>
<TabItem label="TypeScript">
```ts
import { Prompt, type PromptDefinition } from "prompting-press";

const definition: PromptDefinition = {
  name: "summary",
  role: "user",
  body: "Summarise the following article in {{ max_words }} words:\n\n{{ article }}",
  variables: {
    article: { type: "string", origin: "untrusted" },
    max_words: { type: "integer", origin: "trusted" },
  },
  variants: {
    concise: { body: "In one sentence, summarise: {{ article }}" },
    structured: {
      body: "Summarise {{ article }} as a title, three bullets, and a one-line conclusion.",
      meta: { group: "experiment-q4" },
    },
  },
};

const summary = new Prompt(definition);   // same validation as Prompt.fromYaml
Object.keys(summary.variants);            // ["concise", "structured"]
```
</TabItem>
</Tabs>

## Selecting a variant at render

Pass the variant name at render; omit it (or pass `None` / no `variant`) for the default arm. The
resolved name comes back on `RenderResult.variant`.

<Tabs>
<TabItem label="Rust">
```rust
let summary = Prompt::from_yaml(yaml_text)?;

let def = summary.render(&vars, None, &GuardConfig::default())?;          // default (root body)
let concise = summary.render(&vars, Some("concise"), &GuardConfig::default())?;

def.variant;     // => "default"
concise.variant; // => "concise"
```
An unknown variant name returns `Err(ConsumerError::Kernel(..))` with code `"unknown_variant"`.
</TabItem>
<TabItem label="Python">
```python
summary = Prompt.from_yaml(yaml_text)

default = summary.render(Vars, data=data)                    # default (root body)
concise = summary.render(Vars, data=data, variant="concise")

default.variant   # => "default"
concise.variant   # => "concise"
```
An unknown variant name raises `PromptRenderError` with code `"unknown_variant"`.
</TabItem>
<TabItem label="TypeScript">
```ts
const summary = Prompt.fromYaml(yamlText);

const def = summary.render(Vars, data);                       // default (root body)
const concise = summary.render(Vars, data, { variant: "concise" });

def.variant;     // => "default"
concise.variant; // => "concise"
```
An unknown variant name throws `PromptRenderError` with code `"unknown_variant"`.
</TabItem>
</Tabs>

The variant name is a plain string resolved at render time — the set of variants is whatever the
prompt document declares, so it isn't a compile-time enum. It needn't be a magic literal, though: the
names are data you can read off the prompt and an unknown name fails loudly, never silently.

## Discovering the available variants

The `variants` accessor returns the declared variant map; read its keys to see what's selectable (the
default arm is always available and is not listed there — it's the root body, name `"default"`).

<Tabs>
<TabItem label="Rust">
```rust
summary.variants().keys().collect::<Vec<_>>();   // ["concise", "structured"]
summary.variants().contains_key("concise");      // true
```
</TabItem>
<TabItem label="Python">
```python
list(summary.variants)              # ["concise", "structured"]
"concise" in summary.variants       # True
```
</TabItem>
<TabItem label="TypeScript">
```ts
Object.keys(summary.variants);      // ["concise", "structured"]
"concise" in summary.variants;      // true
```
</TabItem>
</Tabs>

Passing a name that isn't declared doesn't fail silently — it returns/raises/throws a structured error
with code `"unknown_variant"` (shown per language above), so a bad selection surfaces immediately.

<Aside type="note" title="Selection is caller-owned">
The library validates the variant name, renders that arm, and stamps provenance — it does **not** own
experiment-assignment logic or pick a variant for you. Which arm to render (an A/B split, a feature
flag, a per-tenant choice) is your application's decision; pass the name you chose.
</Aside>

## Per-variant metadata

Each variant can carry an opaque `meta` map (weights, group labels) for your selection logic — the
library stores and echoes it but never acts on it. See **[Metadata](/prompting-press/guides/metadata/)**
for how `meta` and the prompt-level `metadata` work and how to read them back.
