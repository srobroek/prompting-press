ESM-only · Node 20+ · ships as a native addon (per-platform binary).

`zod` is a peer dependency for the typed-validation facade (the consuming project supplies its own Zod
version). A Zod schema is passed at render time; the library imports only the structural `safeParse`
interface, not Zod's identity, so any object exposing `safeParse` works.

## Write a prompt definition

A prompt definition is YAML, JSON, or TOML text — the same shape in any of the three formats. The
caller reads the text and hands it to the `from*` factories; the library does no file I/O. The
`assistant` definition used throughout this page:

<Tabs syncKey="def-format">
<TabItem label="YAML">
```yaml title="assistant.yaml"
name: assistant
role: system
body: "You are a support assistant for {{ company }}. Keep your replies under {{ max_words }} words."
variables:
  company:
    type: string
    trusted: true
  max_words:
    type: integer
    trusted: true
```
</TabItem>
<TabItem label="JSON">
```json title="assistant.json"
{
  "name": "assistant",
  "role": "system",
  "body": "You are a support assistant for {{ company }}. Keep your replies under {{ max_words }} words.",
  "variables": {
    "company": { "type": "string", "trusted": true },
    "max_words": { "type": "integer", "trusted": true }
  }
}
```
</TabItem>
<TabItem label="TOML">
```toml title="assistant.toml"
name = "assistant"
role = "system"
body = "You are a support assistant for {{ company }}. Keep your replies under {{ max_words }} words."

[variables.company]
type = "string"
trusted = true

[variables.max_words]
type = "integer"
trusted = true
```
</TabItem>
</Tabs>

Construction **validates immediately**: had the body referenced `{{ industry }}` — a variable
`variables` doesn't declare — construction would have **thrown** a `PromptRenderError`, **before any
render** (malformed text throws `LoadError`). A template/variable disagreement is caught at
construction, never as a silent empty render.

### Step 3 — render, and read the result

Hand the `Prompt` from Step 1 the `AssistantVars` schema from Step 2 plus the data — Zod validates, then
the kernel renders:

<Code code={renderAndRead} lang="ts" title="getting-started_typescript_render_and_read.ts" />

`render` has two forms — `variant`/`guard` ride in an optional `opts` object:

```ts
prompt.render(schema, data, opts?);  // schema form: Zod validates data, then renders
prompt.render(data, opts?);          // static form: already-typed data, marshaled directly
// opts: { variant?: string; guard?: GuardConfig | null }
```

`variant` absent selects the default (root body) arm; `guard` absent opts out of the advisory guard.

<Aside type="note">
**TS field naming:** `RenderResult` uses camelCase (`templateHash`, `renderHash`) because napi-rs
maps Rust `snake_case` to JavaScript conventions. The Rust and Python bindings use `template_hash` /
`render_hash`.
</Aside>

## What `RenderResult` carries

| Field | Value |
|-------|-------|
| `text` | The rendered body text. |
| `name` | The prompt name. |
| `variant` | The resolved variant name (`"default"` when none was selected). |
| `templateHash` | `SHA256(resolved variant template source)` — lowercase hex. |
| `renderHash` | `SHA256(rendered output text)` — lowercase hex. |
| `guard` | `string | null` — advisory guard text when opted in; `null` otherwise. |

## Error types

<Code code={errorTypes} lang="ts" title="getting-started_typescript_error_types.ts" />

The error hierarchy (all extend `Error`):

```
PromptingPressError          # base; carries .errors: FieldError[]
├── PromptValidationError    # Zod validation failed      (code "validation")
├── PromptRenderError        # kernel render failure       (codes: unknown_variant,
│                            #   undefined_variable, parse, render, excluded_feature)
└── LoadError                # malformed YAML/JSON/TOML   (code "load")
```

Each `FieldError` carries `field`, `code`, and `message`. A rejected render value is **never**
echoed onto the error surface; a parse error carries template-syntax detail (no bound values).
