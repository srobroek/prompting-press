## ✅ What Went Well (keep doing)

- **The kernel/consumer split paid off exactly as designed.** 004 is the first FFI binding — the whole
  reason 002/003 were built first. The binding is genuinely marshaling + facade only (verified: delegates
  to `prompting_press_core::render` / `prompting_press::{check,get_source,Registry}`, zero engine logic).
  Render parity is structural, not re-tested — Principle I held under real FFI.
- **SEC-004 secret-scrub was designed in, then hardened.** The scrub routed through the consumer's tested
  scrubber from the start; review found two narrow gaps (M-1 raw-pydantic fallback, L-3 model_dump_json
  text) that were closed so the control now holds *by construction*, not just on the happy path.
- **The review pass earned its keep.** review.run + code/security-review surfaced 11 real items, including
  a genuine security-test gap (kernel-path SEC-004 untested from Python) and the I1 CI gap. None were noise.
- **Verify-at-build-time discipline caught a real CI bug.** Trying to actually run the gates exposed that
  `maturin build -m <crate>/Cargo.toml` from repo root mis-derives the module name — the committed T024
  wheel-build path only worked from `packages/python/`. Encoded correctly in the new `ci:test-python` gate.

## ⚠️ What Could Improve (start/stop)

- **The Python test suite shipped un-gated in CI (I1).** The headline differentiator's Python-observable
  proof (validate-in-Python, SEC-004-PY scrub, loader parity) ran only on the dev's machine through T025.
  Caught in code-review, fixed this cycle — but it should have been a task in the original plan, not a
  post-hoc finding. **Start:** when a spec adds a new language package, include "wire its tests into CI" as
  an explicit task, not an implicit assumption.
- **A stale task hint (T006) survived into the code.** tasks.md T006 still says
  `#[pyclass(extends=PyException)]`; the code correctly rejected it (abi3 floor). Harmless, but
  `iterate`/`refine` should have updated the task when the approach changed.
- **quickstart.md had a dict-vs-attribute API mismatch** (`row["field"]` vs `row.field`) — a copy-paste
  trap. **Start:** lint/exec doc code snippets, or generate them from a passing test.

This was escalated in the 002 and 003 retros as "systemic." **This cycle it was total for read-only
audit/research-style agents:** THREE of them returned `tool_uses: 0` and fabricated —

**Counter-observation (important):** the agents that got a WORKING tool channel were *accurate and
valuable* — the 5 review.run agents, the verify agent, and the code/security-review pair all passed their
integrity checks (correct line counts, real file:line citations) and surfaced the 11 genuine findings.

**The discriminator that worked 100% of the time:** a mandatory **integrity check** baked into every
subagent prompt (confirm 2–3 known file sizes / a git stat against expected values; STOP and report
corruption if they don't match). Every glitched agent failed it or returned `tool_uses: 0`; every healthy
agent passed it and was trustworthy. Re-verifying load-bearing findings main-thread before accepting
caught the rest.

  Treat `tool_uses: 0` as automatic "discard + redo main-thread."
