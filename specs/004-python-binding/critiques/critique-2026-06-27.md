# Dual-Lens Critique — Spec 004 (Python binding `prompting-press-py`)

**Date**: 2026-06-27 · **Reviewer**: dual-lens (product strategy + engineering risk) · **Mode**: read-only, pre-implementation
**Artifacts reviewed**: spec.md, plan.md, tasks.md (26 tasks), research.md (D1–D7), data-model.md, contracts/python-api.md, quickstart.md, constitution.md (I–VII), roadmap.md (004 + C-01..C-10)
**Cross-checked against source**: `crates/prompting-press/src/{render,compose,error,registry,check}.rs`, `crates/prompting-press-core/src/engine.rs`, `crates/prompting-press-py/src/lib.rs` + `Cargo.toml`

## Executive Summary

The spec is unusually disciplined: scope is tightly drawn, the four design forks (Q1–Q4) are cleanly clarified, the constitution check is honest, and the boundary defenses (no I/O, no token surface, `pyo3`-only, codegen'd shape) are correctly reproduced from the merged 002/003 work. As a *product* artifact for a developer library it is in good shape — the API contract reads idiomatically and the success criteria are measurable.

The one serious problem is an **engineering grounding gap between the plan/tasks and the actual spec-003 consumer surface they claim to reuse.** The spec's *decisions* are sound (validation owned in Python at the render boundary — Q1; loader reused via FFI — Q3), but the *implementation instructions* in plan.md and tasks.md describe calling consumer functions (`prompting_press::render`, `Composition::resolve`) that **cannot accept the marshaled, already-validated value the Python binding produces** — those functions are generic over `V: Serialize + Validate` and run garde validation internally. The binding must instead call the *kernel* (`prompting_press_core::render`) for the post-validation render path while reusing the *consumer* for loader/registry/check/error. This is recoverable with a plan/tasks correction (no decision reversal), so the verdict is PROCEED WITH UPDATES rather than RETHINK.

**Readiness verdict: PROCEED WITH UPDATES** — resolve E1 (the render-path source mismatch) and E2 (the dual render/compose path is unreflected in the constitution check + C-01 framing) before implementation; the rest are recommendations/questions.

## Product Lens Findings

### P1 🤔 Question — "single idiomatic call path" vs. the two-shaped `render` signature (UX/API)

SC-001 promises "a single idiomatic call path (validate-then-render)" and FR-009 says the binding "MUST NOT require pre-registering a Vars type per prompt." But the contract's `render` signature is overloaded across two argument shapes:

> `vars_model: type[BaseModel], data: dict | <or> vars_instance: BaseModel,` — contracts/python-api.md, "render / get_source"

and tasks T010 spells it `render(reg, name, vars_model_or_instance, data=None, variant=None, guard=None)`. A positional parameter that is *either* a model class (then `data` is required) *or* an already-built instance (then `data` must be absent) is a classic overloaded-signature ergonomics smell in Python — it resists clean type hints and `@overload`, and the failure mode for "passed a class but forgot `data`" / "passed an instance and also `data`" is unspecified. Is the contract finalized as one positional union, or will the plan pick one canonical shape (e.g. always `model + data`, with instances re-validated only when explicitly passed)? Recommend the design doc pin the exact `@overload` set and the misuse error (which `code`?) so US1 is not left to the implementer.

### P2 💡 Recommendation — the headline differentiator (US3 `check`) is sequenced behind two stories from its MVP

The spec is explicit that the lint is the reason the library exists:

> "This is the library's headline differentiator (constitution Principle IV), made runnable from a Python CI pipeline." — spec.md, US3 "Why this priority"

and tasks.md, Notes agrees: "The headline value is US3 (`check()`), but US1 is the MVP that proves the FFI marshaling path." US3 is priority P2, sequenced after US1 (P1) and US2 (P2). That ordering is *defensible* (US1 de-risks the FFI path; `check` needs a populated registry, which needs the loader). But because `check` is what differentiates this library and it depends only on the loader + the consumer's `check` (not on the render-value marshaling that is US1's hard part — see E1), consider whether `check` could ship immediately after US2, or even be demoed earlier with `insert`-only registries. No change required; flag so the sequencing is a conscious product call, not an artifact of priority numbering.

### P3 💡 Recommendation — SC-003 "identical internal representations" is asserted but not observably testable from Python

> "**SC-003**: The same logical prompt loaded from YAML, from JSON, and from a constructed object yields identical internal representations and identical render output" — spec.md, Success Criteria

The Python surface exposes no view of the "internal representation" (the consumer's `PromptDefinition` lives behind the `Registry` `#[pyclass]`; `Registry` has no getter in data-model.md). The quickstart correctly tests the *observable* proxy — identical `render(...).text` across the three forms (quickstart.md US2). Recommend rewording SC-003 to its testable form ("identical render output **and identical provenance** for identical inputs across all three input forms"), since "identical internal representation" is a structural claim the Python tests can't assert and the plan already leans on as structural (Principle I). Minor; prevents a verify-gate ambiguity later.

## Engineering Lens Findings

### E1 🎯 Must-Address — The plan/tasks instruct the binding to call consumer functions that cannot accept the marshaled value (render + compose path mismatch)

This is the load-bearing finding. The clarified decision (Q1: "the binding owns validation at the render boundary," validating in Python via `model_validate` and marshaling an *already-validated* value across FFI) is correct and settled. The problem is that **plan.md and tasks.md then describe reusing consumer entry points that are structurally incompatible with that value shape.**

What the plan/tasks say the binding will call:

> "on success `marshal::to_kernel_value(dumped)` → call `prompting_press::render(def, variant, value, &guard)`" — tasks.md, T010
> "`resolve(reg) -> list[Message]` (**reuses the consumer's `resolve`**...)" — tasks.md, T020
> "the binding calls the consumer's `render`/`get_source`/`check`/`Composition`" — plan.md, Primary Dependencies

What the consumer actually exposes (verified in source):

- `crates/prompting-press/src/render.rs:72` — `pub fn render<V>(reg, name, vars: &V, variant, guard) where V: Serialize + Validate, V::Context: Default`. It runs garde validation *internally* (`vars.validate().map_err(...)?`, line 92) and bridges with `minijinja::Value::from_serialize(vars)` (line 99). There is **no** parameter for a pre-built value.
- `crates/prompting-press/src/compose.rs:144` — `pub fn append<V>(... vars: &V ...) where V: Serialize + Validate` — same internal garde validation (line 156) + `from_serialize` (line 163). `resolve` (line 197) renders the stored, eagerly-validated values.
- Confirmed there is **no** consumer `pub fn` taking a pre-built `minijinja::Value` (only `render<V>` / `get_source` in render.rs; `load_json`/`load_yaml`/`insert`/`get` in registry.rs).

The Python binding, by Q1, produces a type-erased value (`depythonize` → `minijinja::Value`, per research.md D2 / tasks T005). That value is **not** a `V: Serialize + Validate`, so `prompting_press::render(...)` and `Composition::append/resolve` **cannot be called with it.** Calling `render<V>` would also *re-run garde validation* — which the binding deliberately does not want (it already validated via Pydantic; there is no garde model here at all).

The kernel, by contrast, is exactly the right level: `crates/prompting-press-core/src/engine.rs:154` — `pub fn render(def, variant, values: minijinja::Value, guard) -> Result<RenderResult, KernelError>` takes a pre-built value and does not validate. The consumer re-exports the kernel as `prompting_press::core` (`crates/prompting-press/src/lib.rs:176`).

**Consequence:** the binding's render and composition-resolve paths must call `prompting_press_core::render` (the kernel) directly, then map `KernelError` → the exception hierarchy — while still reusing the **consumer** for the loader/registry (`Registry::load_yaml/load_json`), `check`, `CheckReport`/`Finding`/`FindingKind`, `get_source` (which has no `V` and is reusable as-is), and the `error::code` vocabulary. T010 ("call `prompting_press::render`"), T020 ("reuses the consumer's `resolve`"), and the plan's Primary-Dependencies/Summary framing are all inaccurate as written and will not compile against the merged consumer.

**Suggested fix (no decision reversal):** correct T010/T020/plan to: validate in Python → `marshal::to_kernel_value` → `prompting_press_core::render(def, variant, value, &guard)` (kernel) for render; for composition, marshal each entry's validated value and call the kernel per entry in order (re-implementing the trivial in-order loop, OR adding a `Vec<minijinja::Value>` resolve helper) — since `Composition::append<V>`/`resolve` cannot be reused. Keep `KernelError`→exception mapping (the consumer's `From<KernelError>` scrub logic in error.rs is reusable conceptually but is `ConsumerError`-shaped — see E3).

### E2 🎯 Must-Address — The Constitution Check (C-01) and the SEC-004 plan assume "errors arrive pre-scrubbed from the consumer," which is false on the kernel-direct render path

The plan and tasks repeatedly assert the binding only ever sees the consumer's *already-scrubbed* rows:

> "**SEC-004**: surface the consumer's ALREADY-scrubbed `FieldError` rows verbatim — never reach past the consumer to raw `KernelError` detail." — tasks.md, T006
> "the consumer ALREADY scrubs ... The binding surfaces the consumer's already scrubbed `FieldError` rows verbatim — it MUST NOT reach past the consumer to the raw `KernelError` detail." — research.md, D4 SEC-004

But per E1, the render and compose paths call `prompting_press_core::render` directly and therefore receive a **raw `KernelError`** (whose `Parse`/`Render` `detail` *does* embed bound-value content — confirmed in `crates/prompting-press/src/error.rs:178-209` and its scrub tests). On those paths the binding **must perform the SEC-004 scrub itself** (or route the `KernelError` through the consumer's `From<KernelError> for ConsumerError` impl, which is the merged, tested scrubber). The current tasks tell the implementer the opposite — "never reach past the consumer to raw `KernelError`" — which is impossible to honor when the binding is the thing calling the kernel. This is a security-relevant instruction error: an implementer following T006 literally on a kernel-direct path could surface unscrubbed detail. The T008 scrub test (`ConsumerError::Kernel(Render{detail with a seeded secret}) → PromptRenderError whose str() ... do NOT contain the secret`) also presumes a `ConsumerError`, not the raw `KernelError` the kernel path yields.

**Suggested fix:** make the render/compose error path route `KernelError` → `ConsumerError` via the consumer's existing `From<KernelError>` impl (reuse the tested scrubber — `prompting_press::ConsumerError::from(kernel_err)`), then `ConsumerError` → exception hierarchy. Then the "surface the consumer's already-scrubbed rows" framing becomes true again. Update T006/T008/research D4 to say the binding *applies the consumer's `From<KernelError>` normalizer* on the kernel-direct path, rather than claiming the kernel is never directly called.

### E3 💡 Recommendation — The `ConsumerError` → exception map is 1:1 by *variant*, but `Validation`/`Kernel` both carry `Vec<FieldError>` with heterogeneous codes; the subtype split needs a rule

The plan maps "1:1 onto the Rust `ConsumerError` variants" (plan.md Constitution Check; D4). The actual enum (`crates/prompting-press/src/error.rs:100`) has four variants: `Validation(Vec<FieldError>)`, `Kernel(Vec<FieldError>)`, `UnknownPrompt(String)`, `Load(String)`. The exception hierarchy has four subtypes — but `PromptRenderError` is documented to cover `unknown_variant | undefined_variable | parse | render | excluded_feature` (data-model.md, contracts), which all live inside the single `ConsumerError::Kernel` variant and are distinguished only by the per-row `code` string. So the map is **variant→subtype for three variants, but `Kernel`→`PromptRenderError` regardless of inner code** — and `UnknownPrompt` is a *variant* (not a `code`-bearing `FieldError` row at all; it's a bare `String`). The implementer needs an explicit rule for: (a) what `.errors` rows `UnknownPromptError`/`LoadError` carry given the source has no `FieldError` for those variants (data-model.md says `UnknownPromptError` "carries the requested name" — as a row? with what `field`/`code`?); (b) confirmation that `unknown_variant` lands in `PromptRenderError` not `UnknownPromptError` (easy to conflate by name). Recommend the design doc add the exhaustive `ConsumerError` variant + inner-`code` → exception-subtype + `.errors`-rows table. The "EXHAUSTIVE match → compile error on a new variant" intent in T006 is good and should be kept.

### E4 💡 Recommendation — `model_dump(mode=...)` choice is left open but silently changes cross-binding marshaling parity (spec 006)

research.md D2 leaves the dump mode unresolved: "confirm at impl whether to dump with `mode="json"` ... vs `mode="python"`. **Lean `mode="json"**`." T005 says "Decide & pin the `model_dump` mode at the call site (lean `mode="json"`...)." This is the right lean, but it is more load-bearing than "an impl detail": `mode="json"` stringifies `datetime`/`Decimal`, `mode="python"` keeps them as Python objects, and `depythonize` will then produce *different* `minijinja::Value`s — which directly determines whether this binding's marshaling matches the TS (napi) binding's in the spec-006 conformance corpus (the corpus's whole point per roadmap C-07). Since 004 explicitly defers the corpus ("the broad cross-binding corpus is spec 006" — spec.md Edge Cases), the risk is that 004 pins a mode in isolation that 006 later finds disagrees with the TS binding. Recommend: (a) pin `mode="json"` now *and* record the date/Decimal wire shape (ISO-8601 string? Decimal-as-string?) as an explicit cross-binding contract note in data-model.md so spec 005/006 inherit it rather than rediscover a conflict. Low cost now, expensive to reconcile post-005.

### E5 🤔 Question — `Composition.from_messages` takes Pydantic instances, but `render` is moving toward `model + data`; which validation entry does compose use?

The compose contract is `Composition.from_messages([(name, vars, variant?), ...])` and `append(name, vars, variant=None)` with "eager-validate" (contracts; T020; research D7). The quickstart passes **already-constructed instances**: `("greet", Greeting(name="Ada", count=3), None)` (quickstart.md US4). But Q1/FR-002 says the binding *owns validation at the render boundary* and re-validates instances. If compose entries are constructed instances, does `append` re-run `model_validate` on each (consistent with "the binding owns validation"), or does it trust the already-constructed instance (in which case a caller who built an instance bypassing validators — e.g. `model_construct`) gets an unvalidated render)? The eager-validate decision (option (a), D7) implies re-validation, but the quickstart's instance-passing example doesn't show the `model + data` shape that US1's `render` is debating in P1. Recommend the contract state explicitly that `append`/`from_messages` re-validate passed instances (same rule as `render`), so the "validation always runs" guarantee (compose.rs module-doc invariant) holds across the FFI boundary too.

### E6 💡 Recommendation — abi3-py310 floor reconciliation is correct; add a guard so the three-way agreement can't silently re-drift

The Q4 fix is sound and well-justified (the `abi3-py39` + `X | None`-codegen mismatch was a real install-then-ImportError trap — research D1; spec Assumptions). T001 bumps the crate feature and T002 leaves `requires-python = ">=3.10"`. The latent risk that *created* this bug — three independent declarations of the floor (crate feature, `requires-python`, codegen `--target-python-version`) with nothing enforcing agreement — is fixed in value but **not** guarded against recurrence. Recommend a tiny CI assertion (or a note in T025's gate suite) that the three floors agree, so a future `abi3` bump or codegen-target change can't reopen the same trap. The watch-item (CPython 3.10 EOL 2026-10-31, plan.md "Verified-this-cycle") is correctly noted and correctly deferred to spec 007; no action now.

## Cross-Lens Insights

- **E1 + E2 are one root cause with two symptoms.** Because the plan modeled the binding as "a thin wrapper that calls the consumer's `render`/`compose`," it inherited the consumer's "errors are pre-scrubbed" property for free. Once you correct E1 (render/compose go to the *kernel*, since the consumer's generic-over-`V` signatures can't take a marshaled value), the pre-scrubbed assumption (E2) breaks on the same paths. Fixing both with one move — route the kernel-direct render's `KernelError` through the consumer's existing `From<KernelError>` normalizer — preserves the SEC-004 guarantee *and* keeps the "no scrub logic in the binding" spirit of C-02. This should be the first design correction.
- **The constitution-compliance story is genuinely strong and should not be over-corrected.** C-02 (`pyo3`/`pythonize` only here), C-03 (no token surface — F4 correctly applied; roadmap reconciliation flagged in Assumptions), C-07 (codegen'd shape, freshness-gated) are all correctly reproduced. E1/E2 do **not** threaten C-01: calling the *kernel* directly is still "no engine logic in the binding" — rendering still happens once, in Rust. But the plan's C-01 row ("the binding MARSHALS to the consumer's `render`...") should be reworded to "marshals to the kernel's `render` (re-exported via `prompting_press::core`) and the consumer's loader/check" so the Constitution Check matches what will actually be built. That is the only constitution-table edit E1 forces.
- **Product framing is ahead of the engineering plan's accuracy.** The spec (intent) is internally consistent and well-scoped; the *plan/tasks* (mechanism) drifted from the merged source they depend on. This is the expected failure mode when 004 was specced "mirroring 003" without re-reading 003's exact generic signatures — and it is exactly what a pre-implementation critique should catch.
