# Dual-Lens Critique — Spec 005 (TypeScript binding `prompting-press-node`)

**Date**: 2026-06-27 · **Reviewer**: dual-lens (product strategy + engineering risk) · **Mode**: read-only, pre-implementation
**Artifacts reviewed**: spec.md, plan.md, tasks.md (31 tasks), research.md (D1–D7), data-model.md, contracts/ts-api.md, quickstart.md, checklists/binding.md, constitution.md (I–VII), roadmap.md (005 + C-01..C-10)
**Cross-checked against source**: `crates/prompting-press/src/{render,compose,error,check}.rs`, `crates/prompting-press-core/src/engine.rs`, `crates/prompting-press/src/lib.rs`, `crates/prompting-press-node/{Cargo.toml,src/lib.rs}`, `packages/typescript/package.json`, the merged spec-004 binding.

## Executive Summary

Spec 005 is in **stronger** shape pre-implementation than 004 was, precisely because it learned from
004's critique: the load-bearing engineering finding that 004 had to fix mid-flight (render/compose must
call the *kernel* directly, since the consumer's `render<V>`/`append<V>` are generic over a garde
`Validate` type — and the SEC-004 scrub must therefore run on the kernel-direct path) is **already baked
into the 005 plan and tasks** (plan §Technical approach; T011/T022 "call `prompting_press_core::render`
DIRECTLY"; T006 "route the raw `KernelError` through the consumer's `From<KernelError>` scrubber FIRST").
I re-verified the consumer source: `render<V>`/`append<V>` are still generic (render.rs:72, compose.rs:144),
the kernel is re-exported as `prompting_press::core` (lib.rs:176), and `ConsumerError`'s `From<KernelError>`
still scrubs — so the carried-forward design is correct against the merged code.

The remaining findings are TS-ecosystem-specific and none is a decision reversal. The one genuinely new
structural risk is **the Rust-addon / TS-facade split** (Zod can't live in Rust, so validation + the
`Error` hierarchy live in `packages/typescript/src/index.ts` over the napi addon) — sound, but it widens
the binding's surface and needs the napi error-return mechanism pinned so structured rows survive the
addon→facade hop.

**Readiness verdict: PROCEED** — no Must-Address blockers (the 004 E1/E2 traps are pre-resolved). Address
the recommendations (esp. E1: pin the napi error-return mechanism) during implementation.

## Product Lens Findings

### P1 💡 Recommendation — the headline differentiator (US3 `check`) is again sequenced behind US1+US2

Same observation as 004 P2 (and the same defensible answer): `check` is the library's reason to exist
(spec §US3 "Why this priority") but is P2 after US1 (P1) + US2 (P2). It depends only on the loader +
consumer `check`, not on US1's value-marshaling. The sequencing is a conscious call (US1 de-risks the
napi path; `check` needs a populated registry). No change required — flagged so it stays deliberate.

### P2 💡 Recommendation — SC-003 "identical internal representations" is asserted but not observable from TS

Carried from 004 P3 and unchanged here: the TS surface exposes no view of the consumer's internal
representation; the quickstart correctly tests the observable proxy (identical `render().text` + identical
`templateHash`/`renderHash` across YAML/JSON/object). Recommend SC-003 lean on its testable form
("identical render output **and provenance**") — minor, prevents a verify-gate ambiguity. (Inherited from
the 004 SC text the 005 spec mirrors.)

### P3 🤔 Question — does `render` accept a Zod schema, plain data, or both, and how is the misuse surfaced?

FR-001 + clarified Q4 say both a Zod schema (validated via `safeParse`) and plain already-typed data are
accepted. The contract (`render(reg, name, schema, data, opts?)`) implies schema-then-data, but the
plain-data path's shape (is it `render(reg, name, data)` with no schema arg? an overload? a separate
entry?) is not pinned in `contracts/ts-api.md`. This is the TS analogue of 004's P1 overloaded-signature
smell. Recommend the contract pin the exact TS `@overload`/union and the misuse behavior (e.g. a non-Zod,
non-plain value) before US1, so the implementer isn't left to invent it.

## Engineering Lens Findings

### E1 🎯→💡 (pre-resolved structurally; one open sub-item) — the napi error-return mechanism must be pinned so structured rows survive the addon→facade hop

The 004 Must-Address pair (E1 kernel-direct render; E2 SEC-004 scrub on that path) is **already correct in
005** (verified above). What 005 adds — and what is *not yet pinned* — is the **addon→facade error
channel**. Because the `Error` subclasses + the row-to-subclass mapping live in TS (Zod can't live in
Rust), the napi addon must hand the TS facade the **scrubbed `[{field,code,message}]` rows + the variant
discriminator** in a *structured* form, not a flattened string. research.md D4 flags this ("pick the one
that keeps rows structured, not string-encoded if avoidable") but leaves it open. If the implementer
takes the easy path (encode the rows into a `napi::Error` message string), the TS facade has to re-parse a
string — fragile and a SEC-004 risk if the raw message ever includes detail. **Recommend pinning, before
US1 (T006/T011/T013): the addon returns errors as a structured payload** (e.g. a discriminated result
object the facade maps, or a typed `napi` error object carrying the rows array), so the TS facade
constructs the right subclass from structured data. This is the 005 equivalent of getting the error path
right up front — downgraded from Must-Address only because the *scrub* (the security-critical part) is
already correctly placed in Rust (T006).

### E2 💡 Recommendation — the Rust-addon / TS-facade split needs the "which layer validates compose entries" rule pinned (004 E5 analogue)

`Composition.fromMessages`/`append` (T022) take (name, vars, variant) entries. Per Q1 the binding owns
validation at the boundary — but with the split, *where* does a compose entry's Zod `safeParse` run? The
plan says "Vars validation for compose entries lives in the TS facade `fromMessages`/`append` wrapper,
mirroring US1's `safeParse`-at-boundary" (T022 parenthetical) — good, but the napi `Composition.append`
also exists and receives an already-validated value. The contract should state explicitly that the
**TS-facade** `Composition` is what callers use (it `safeParse`s each entry, then feeds the validated
value to the napi addon), so a caller can't reach the napi `Composition.append` and bypass validation.
This is the 004 E5 question, sharpened by the two-layer split. Recommend the contract/data-model pin that
the public `Composition` is the TS-facade one.

### E3 💡 Recommendation — `ConsumerError` → `Error`-subclass map needs the same exhaustive table 004 needed (E3 analogue)

Identical to 004 E3: `ConsumerError` has 4 variants (`Validation(Vec<FieldError>)`, `Kernel(Vec<FieldError>)`,
`UnknownPrompt(String)`, `Load(String)` — verified error.rs), but the `Error` hierarchy has 4 subclasses
where `Kernel` covers 5 inner `code`s (`unknown_variant`/`undefined_variable`/`parse`/`render`/
`excluded_feature`) and `UnknownPrompt`/`Load` are bare strings with no `FieldError` rows. The mapper
(T006 Rust-side discriminator + T008/T013 TS-side subclass construction) needs the exhaustive
variant + inner-`code` → subclass + `.errors`-rows table, incl. what rows `UnknownPromptError`/`LoadError`
carry (field/code for a bare-String variant). Recommend adding it to data-model.md or the contract.
(004 resolved this in its data-model; 005 should inherit the same table.)

### E4 💡 Recommendation — pin the JS↔serde marshaling primitive + bigint/date wire shape now (cross-binding parity, 004 E4 analogue)

research.md D2 leans on napi's serde support (JS value → `serde_json::Value` → `minijinja::Value::from_serialize`)
and the Q6 null/undefined rule — correct and matched to the Python binding. But two wire-shape decisions
are left to impl: **bigint** (does napi's bigint→serde preserve it, or lossy-cast to f64?) and **dates**
(JS `Date` → what serde shape? — the Python binding pins ISO-8601-string via `mode="json"`). Because spec
006's conformance corpus compares *this* binding's marshaling to the Python binding's, a mismatch here is
exactly what 006 exists to catch — cheaper to pin now. Recommend data-model.md record the bigint + date
wire shape as a cross-binding contract note, matched to 004's date/Decimal-as-string decision. (This is
the single most important "pin it now" item for the spec-006 dependency.)

### E5 💡 Recommendation — the three new CI gates (T028/T029/T030) are correctly planned; verify the napi Linux link path early

The plan carries the 004 lessons well: T028 extends `ci:check-ffi` to `napi`, T029 adds the Node advisory
gate, T030 adds `ci:test-node` and explicitly says "verify the napi build + addon-load works on the LINUX
runner (the maturin/libpython lesson)." This is the right set. One sharpening: the 004 `ci:test-python`
gate *failed first in CI* on exactly this class of bug (libpython not found at runtime on Linux). Recommend
T030 be sequenced/verified **early** (not left to the end of Polish), since a napi addon-load failure on
Linux would block the whole TS test suite and is the highest-probability CI surprise. Consider a smoke
"build + `import` on Linux" check as soon as the addon has any `#[napi]` export (after Foundational),
rather than discovering it at T030.

### E6 🤔 Question — Rust-side unit tests for a napi addon: feasible, or TS-only?

T009 calls for Rust-side `#[cfg(test)]` marshaling/scrub tests in the node crate. PyO3 supports
`Python::attach` in `cargo test`; napi's test story is different (the addon usually needs a Node runtime).
Confirm whether the scrub/marshaling logic can be unit-tested in pure Rust (it should — the scrub is
`ConsumerError`-shaped and the marshal is serde, neither needs a live JS engine), or whether those tests
must move to the TS side. Recommend research.md D6/T009 confirm the napi `cargo test` harness boundary so
T009 isn't written against an unavailable `Env`. (Low risk — the scrub + serde marshal are JS-engine-free —
but worth confirming so the test split is right.)

## Cross-Lens Insights

- **005's plan is accurate against the merged source where 004's was not.** The 004 critique's root cause
  was specced "mirroring 003" without re-reading 003's generic signatures; 005 was specced *after* 004
  shipped and explicitly carries the kernel-direct + scrub-on-kernel-path corrections. I re-verified the
  signatures — the carry-forward is correct. The pre-implementation critique's job here is mostly
  *confirming the inheritance* + pinning the new TS-specific seams (E1 error channel, E4 wire shape).
- **The one truly new thing is the two-layer split** (napi addon + TS facade). It's sound and idiomatic
  (Zod is a TS library), preserves zero-engine-logic, and the plan/tasks model it correctly (T011 addon
  render + T013 facade `safeParse` wrapper). But it doubles the error-path hops, so E1 (structured
  addon→facade error channel) and E2 (which layer's `Composition` is public) are the seams to pin so the
  split doesn't leak a native type or bypass validation.
- **Constitution compliance is strong and should not be over-corrected.** C-02 (napi only in `-node`, gate
  extended — T028), C-03 (no token surface — F4, already reconciled for 005 unlike 004), C-07 (codegen'd
  TS shape, freshness-gated) all correctly reproduced. None of the findings threatens C-01: calling the
  kernel directly is still "no engine logic in the binding."
