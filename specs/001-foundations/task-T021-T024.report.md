## T021 — Pin + determinism + lockfile

### Pinned versions (exact, no `^`/`~`)
`packages/typescript/package.json` devDependencies:
```json
"@napi-rs/cli": "3.7.2",
"json-schema-to-typescript": "15.0.4",
"prettier": "3.8.4",
"typescript": "5.9.2"
```
Runtime `dependencies` stays empty (json2ts + prettier are build/dev tools only). `"private": true` left as-is (spec 007 handles publishing; does not block codegen/install).

**Why prettier is pinned explicitly:** json2ts 15.0.4 declares prettier as a *regular* dependency (`prettier: ^3.2.5`) and uses it internally to format output. An unpinned prettier ⇒ nondeterministic formatting across machines. Pinning prettier 3.8.4 as a devDep forces pnpm to resolve a *single* prettier instance. Lockfile confirms one resolution only: `prettier@3.8.4` (no duplicate node).

### Lockfile (SEC-001/002 supply-chain pin)
- `mise exec -- pnpm install` (pnpm 10.33.2) run in `packages/typescript/`.
- `pnpm-lock.yaml` generated and **committed** (tracked, not gitignored — `git check-ignore` returns exit 1).
- 123 `integrity:` hashes present; `lockfileVersion: '9.0'`.
- CI uses `pnpm install --frozen-lockfile` against this lockfile.

### Determinism flags
- `--bannerComment ''` empties json2ts's built-in banner. (Note: json2ts 15.0.4's default banner is already static — no version/timestamp — but the empty banner + our own banner gives full control.)
- A **static** "GENERATED — DO NOT EDIT" banner is prepended by the codegen wrapper. It carries **no version, no timestamp, no host data**.
- Output normalized to LF line endings + exactly one trailing newline.

### Exact json2ts invocation (recorded for T026)
Canonical form is the wrapper script `packages/typescript/scripts/codegen.mjs`, which calls `compileFromFile` programmatically because it also owns the banner + newline normalization:
```js
compileFromFile(SCHEMA_PATH, {
  bannerComment: "",
  additionalProperties: false,   // sealed objects stay closed (no [k:string] index sig)
  unknownAny: true,              // emit `unknown` instead of `any`
  cwd: dirname(SCHEMA_PATH),
});
```
Equivalent raw CLI (same flags; the wrapper additionally fixes banner + newline):
```
json2ts \
  --input  ../../schemas/jsonschema/prompt-definition.schema.json \
  --output src/generated/prompt-definition.ts \
  --bannerComment '' \
  --no-additionalProperties \
  --unknownAny
```
`package.json` script: `"codegen": "node ./scripts/codegen.mjs"`. T026 reuses it via `pnpm -C packages/typescript codegen`.

Sealed object as a CLOSED interface — confirmed **no** `[k: string]: unknown` index signature (the schema's `additionalProperties: false` + json2ts `additionalProperties:false` keep it closed):
```ts

  body: string;
```

(`PromptDefinition` and `VariableDecl` are likewise closed — no index signature.)


Open record maps (`variables`/`variants`, the `additionalProperties: { $ref }` pattern — these are *value-typed maps*, distinct from sealed objects):
```ts

variants?:  { [k: string]: Variant };
```
The `propertyNames` "not const default" constraint on `variants` is correctly dropped from the type (enforced by the US2 validation gate, not the type).

## tsc type-check
`mise exec -- pnpm -C packages/typescript exec tsc --noEmit` → **exit 0**.

`tsconfig.json` `include` was changed from `[]` to `["src/generated/**/*.ts"]`. Rationale: the original empty `include` made `tsc` fail with TS18003 ("No inputs were found") — there genuinely was no source. Now that the committed generated shape exists, pointing `include` at it makes the strict config actually type-check the generated file (the local proof US4 wants). The thin wrapper source lands in a later spec and will widen `include` then. The generated file also type-checks standalone under all strict flags (`--strict --noUncheckedIndexedAccess --exactOptionalPropertyTypes` etc.), exit 0.
