### src/lib.rs

```rust
//! # prompting-press-py
//!
//! The Python binding for Prompting Press, built with [PyO3]. This crate exposes the
//! Rust consumer surface ([`prompting_press`]) to Python as a native extension module.
//!
//! It is one of exactly two crates (the other being `prompting-press-node`) permitted
//! to depend on an FFI toolkit; the kernel and the Rust consumer stay FFI-free
//! (constitution Principle II / C-02).
//!
//! This is a spec-001 stub: it registers a single trivial function to prove the PyO3
//! module wiring works end to end.
//!
//! [PyO3]: https://pyo3.rs

use pyo3::prelude::*;

/// Returns the kernel version, reached through the Rust consumer surface.
///
/// Placeholder so the extension module exports something callable from Python and so
/// the dependency edges onto `prompting-press`/`prompting-press-core` are exercised.
#[pyfunction]
fn core_version() -> &'static str {
    prompting_press::core_version()
}

/// The native module Python imports as `prompting_press_py`.
#[pymodule]
fn prompting_press_py(m: &Bound<'_, PyModule>) -> PyResult<()> {
    m.add_function(wrap_pyfunction!(core_version, m)?)?;
    Ok(())
}
```

## T014 link fix — `prompting-press-py` cdylib link failure on macOS

### Problem (found by review gate)

T005–T012 verification used `cargo check --workspace`, which type-checks but does **not**
link. T014 requires `cargo build --workspace` to build **and link** all four crates. A
full build fails at the link step on **only** the `prompting-press-py` cdylib:

```
            "__Py_NoneStruct", referenced from:
                pyo3_ffi::object::Py_None::... in libpyo3-...rlib(...)
          ld: symbol(s) not found for architecture arm64
          clang: error: linker command failed with exit code 1
error: could not compile `prompting-press-py` (lib) due to 1 previous error
```

`prompting-press-core`, `prompting-press`, and `prompting-press-node` build+link cleanly;
only the pyo3 crate fails.

### Root cause

PyO3's `extension-module` feature deliberately leaves CPython symbols
(`__Py_NoneStruct`, `_Py_IncRef`, `_Py_InitializeEx`, ...) **undefined**, for the Python
interpreter to resolve when it loads the extension. A bare `cargo build` of the
standalone cdylib has no interpreter to link against. On macOS the linker rejects the
undefined symbols unless told to defer them (`-undefined dynamic_lookup`); Linux ELF
permits undefined symbols in a shared object, so it needs nothing. This is expected,
documented PyO3 behavior — not a misconfiguration.

### Fix — crate-scoped `build.rs` (not repo-wide `.cargo/config.toml`)

`crates/prompting-press-py/build.rs`:

```rust
fn main() {
    // PyO3's `extension-module` feature leaves CPython symbols (`__Py_NoneStruct`,
    // `_Py_IncRef`, ...) undefined for the interpreter to resolve at load time. A bare
    // `cargo build` of this standalone cdylib has no interpreter to link against, so on
    // macOS the linker (unlike Linux ELF, which permits undefined symbols in a dylib)
    // needs `-undefined dynamic_lookup` to defer their resolution to load time.
    //
    // Scoped here via `cargo:rustc-link-arg` (not a repo-wide .cargo/config.toml) so the
    // flags attach ONLY to this crate's link phase and never enter the RUSTFLAGS
    // fingerprint — keeping the US3 codegen-determinism gate and the US4 `cargo tree`
    // FFI-isolation gate unperturbed. Linux/Windows need nothing.
    if cfg!(target_os = "macos") {
        println!("cargo:rustc-link-arg=-undefined");
        println!("cargo:rustc-link-arg=dynamic_lookup");
    }
}
```

**Why `build.rs` `rustc-link-arg` over `.cargo/config.toml` rustflags:** a build-script
`cargo:rustc-link-arg` affects only this crate's link phase and does **not** enter the
RUSTFLAGS fingerprint, so it cannot perturb codegen determinism (US3) or the `cargo tree`
dependency metadata the FFI-isolation gate reads (US4). A workspace-wide rustflags config
would touch both surfaces. The `target_os = "macos"` guard keeps Linux/Windows builds
flag-free.

The `extension-module` / `abi3-py39` wiring in `crates/prompting-press-py/Cargo.toml` is
unchanged, so it stays consistent with `packages/python/pyproject.toml`'s maturin
`features = ["pyo3/extension-module"]`. The fix is purely additive (link args) and
touches only files inside `crates/prompting-press-py/`.
