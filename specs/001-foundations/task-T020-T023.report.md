## T020 — Exact codegen invocation (all flags)

Recorded canonically in `packages/python/scripts/codegen.sh` (single source; **T026 freshness gate re-runs this script**). The literal invocation:

```bash
mise exec -- datamodel-codegen \
  --input schemas/jsonschema/prompt-definition.schema.json \
  --input-file-type jsonschema \
  --output-model-type pydantic_v2.BaseModel \
  --target-python-version 3.10 \
  --use-annotated \
  --disable-future-imports \
  --disable-timestamp \
  --formatters builtin \
  --custom-file-header "$HEADER" \
  --output packages/python/python/prompting_press/generated/prompt_definition.py
```

### Flag rationale

| Flag | Why |
|---|---|
| `--input-file-type jsonschema` | Source is Draft 2020-12 JSON Schema. |
| `--output-model-type pydantic_v2.BaseModel` | TRUE Pydantic v2 (`model_config = ConfigDict(...)`), not v1 `class Config`. |
| `--target-python-version 3.10` | Matches `packages/python/pyproject.toml` `requires-python = ">=3.10"`; fixes typing syntax (PEP 604 `X | Y`, builtin generics `dict[str, Any]`) so output doesn't vary by interpreter. |
| `--disable-timestamp` | Removes the generation timestamp from the header — the primary determinism leak. |
| `--formatters builtin` | Decouples output from installed **black/isort versions**. datamodel-codegen's default formatters are black+isort; their formatting output is version-dependent, so a black upgrade would churn the generated file across machines. `builtin` uses the in-tool formatter → no external formatter-version dependency. (This is the path datamodel-codegen's own FutureWarning points to.) |
| `--custom-file-header "$HEADER"` | Injects an explicit, deterministic "GENERATED — DO NOT EDIT" header (see below). |
| `--use-annotated` | **Correctness, not just style.** Emits `Annotated[str, Field(min_length=1)]` / `Annotated[int, Field(ge=0)]` instead of the `constr(min_length=1)` / `conint(ge=0)` callables. The callable form is unresolvable as a deferred annotation and breaks Pydantic model build. Implies `--field-constraints`. |
| `--disable-future-imports` | Drops `from __future__ import annotations`. Eager annotations make the model build robust under both normal import AND the `importlib.spec_from_file_location` synthetic-module load (the exact pattern the verification command uses); deferred annotations otherwise fail Pydantic's forward-ref resolution under a synthetic module name. Safe on the 3.10 floor because `X \| Y` and `dict[str, ...]` are valid at runtime in 3.10+. |

### Header-version churn (explicitly handled)

datamodel-code-generator **0.65.1** makes the version line, command-line header, and `@generated` marker **opt-in** (`--enable-version-header`, `--enable-command-header`, `--enable-generated-header-marker`). None are passed, so there is **no tool-version string** in the output to churn on a tool upgrade. The header is fully controlled by `--custom-file-header`. (`--disable-warnings` would not have affected the header; it was not needed.)

## T020 — Hash-pin mechanism (SEC-001/002)

**Chosen: uv-native — a PEP 735 `[dependency-groups]` + a committed `uv.lock`.**

The task text offered `requirements-codegen.txt` + `--require-hashes` *or* a uv-managed lock as options. **uv.lock was chosen for repo consistency**: this repo is uv-based (uv 0.11.8 pinned in `mise.toml`, used throughout). A standalone hash-pinned `requirements.txt` would introduce a second, redundant dependency-management mechanism. (A working `requirements-codegen.txt` was built and verified during exploration, then **deleted** so only one mechanism — `uv.lock` — exists.)

Mechanism:

1. **`packages/python/pyproject.toml`** declares the codegen tool as a build-time-only dependency-group; runtime `dependencies` stays `[]`:
   ```toml
   [dependency-groups]
   codegen = ["datamodel-code-generator==0.65.1"]
   ```
   The version must match the `mise.toml` pin (`pipx:datamodel-code-generator`); bump both together, then re-lock.

2. **`packages/python/uv.lock`** (committed) is the SEC-001/002 supply-chain pin. uv.lock carries package integrity hashes **natively** — no `--generate-hashes`/requirements.txt needed. Facts about the committed lock:
   - **383 `sha256` integrity hashes** across the full transitive closure (25 resolved packages including the project).
   - `datamodel-code-generator == 0.65.1` recorded under `[package.dev-dependencies] codegen` with `specifier = "==0.65.1"`.
   - black/isort appear in the closure (declared deps of datamodel-code-generator) but are **never invoked** at codegen time because `--formatters builtin` bypasses them; they are hashed only for closure completeness.

   Re-lock with:
   ```bash
   (cd packages/python && mise exec -- uv lock)
   ```

### How CI uses it (hermetic, hashed)
```bash
# 1. Install the codegen group from the lock. --no-install-project avoids building
#    the maturin extension (the binding crate doesn't exist yet in the skeleton);
#    --frozen asserts the lock is up to date with pyproject.toml.
(cd packages/python && mise exec -- uv sync --group codegen --no-install-project --frozen)
# 2. Generate (canonical invocation lives in the script):
mise exec -- bash packages/python/scripts/codegen.sh
```
**Verified:** `uv sync --group codegen --no-install-project --frozen` installs the group hermetically from the lock (exit 0), and `uv run --group codegen --no-sync datamodel-codegen --version` reports `0.65.1`. The mise pin remains the dev convenience; `uv.lock` is the hermetic, hashed CI source.

### Key excerpts — Pydantic v2 fidelity

**Imports** (true v2, no `constr`/`conint` callables, no `__future__`):
```python
from enum import Enum
from typing import Annotated, Any
from pydantic import BaseModel, ConfigDict, Field
```

**Enums** for `role` and `provenance` (plus the two type-keyword enums):
```python
class Role(Enum):
    system = 'system'; user = 'user'; assistant = 'assistant'

class Provenance(Enum):
    trusted = 'trusted'; untrusted = 'untrusted'; external = 'external'
```

**Sealed object with `extra='forbid'`** (all three sealed models — `VariableDecl`, `Variant`, `PromptDefinition` — carry it):
```python
class PromptDefinition(BaseModel):
    model_config = ConfigDict(
        extra='forbid',
    )
    name: Annotated[str, Field(description="...", min_length=1)]
    role: Annotated[Role, Field(description="...")]
    body: Annotated[str, Field(description="...")]
    ...
```

**Open dict fields** — `meta` / `metadata` (library-opaque) are `dict[str, Any] | None`:
```python
metadata: Annotated[dict[str, Any] | None, Field(description='Arbitrary prompt-level metadata; library-OPAQUE ...')] = None
meta:     Annotated[dict[str, Any] | None, Field(description="The default (root) arm's selection metadata ...")] = None
```

**Constraint** preserved via `Annotated` (resolvable, unlike `conint`):
```python
minLength: Annotated[int | None, Field(ge=0)] = None
```

Semantic checks (against the committed file, real-module import, pydantic 2.13.4):
- construct valid `PromptDefinition` → OK; `role` is a `Role` enum instance.
- unknown key (`extra='forbid'`) → `ValidationError` raised. ✅
- `name=''` (`min_length=1`) → `ValidationError` raised. ✅
- open `meta={'weight': 2}` → accepted. ✅
- `PromptDefinition.model_config['extra'] == 'forbid'`, config built via `ConfigDict` (v2). ✅

> Note: `propertyNames` (variants-named-`default` rejection, FR-011b) is silently dropped by datamodel-codegen — by design. That constraint is enforced by the US2 validation gate, not the shape model.
