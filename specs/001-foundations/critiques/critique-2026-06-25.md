## Executive Summary

Spec 001 is **solid and ready to implement**. It has been through an exhaustive grilling session, a

## Product Lens Findings

**3a/3b Problem & value** — The spec solves a real, well-scoped problem (the spine specs 002–007
depend on) and its MVP (US1 layout) is correctly identified. User value is clear *for the eventual
library*, less so *within 001* (nothing renders) — but that's the deliberate roadmap sequencing, not
a flaw. ✓

**3c Alternative / simpler approach — strongest finding (see X1).** Building the codegen pipeline for
**all three** languages in 001, when only Python ships first (Bellwether) and the TS/Rust *bindings*
don't exist until specs 004/005, is partially speculative: the generated TS and Rust shapes have **no
consumer** in 001 — they land in skeleton packages nothing imports. The pipeline + Python target
fully demonstrates "schema is the single source of truth"; the other two targets could be deferred to
their binding specs.

**3d Edge cases** — well covered (the schema reject-matrix, partial-regeneration, parse-vs-schema
invalid). ✓

**3e Success measurement** — SC-001..007 are measurable and each has a verifying task. ✓

## Engineering Lens Findings

**4a Architecture** — sound; the symmetric kernel + per-language-consumer layout is verified-idiomatic
(research), and the FFI-isolation invariant is both stated and CI-enforced. ✓

**4a/E1 Generated artifact inside the hand-written consumer crate.** Plan/tasks put the generated Rust
shape at `crates/prompting-press/src/generated/prompt_definition.rs` — i.e. generated code lives
*inside* the hand-written consumer crate's `src/`. Two consequences: (1) the consumer crate won't
compile until codegen has run (a CI ordering constraint — codegen must precede `cargo build`); (2)
mixing generated + hand-written in one `src/` raises the chance someone hand-edits a generated file.
The freshness gate catches (2), but the ordering (1) should be explicit. *Suggestion: confirm codegen
runs before build in the moon task graph, and keep generated code in a clearly-named `generated/`
module that `lib.rs` re-exports but never edits.*

**4b Failure modes** — minimal for a no-runtime spec. The one real failure mode is **codegen
non-determinism** (formatter/toolchain drift), which research D1 + tasks T020–T022 handle via pinning.
✓ Well-managed.

**4c Security** — N/A. 001 processes no input, makes no calls, handles no untrusted data (provenance
is only *declared* in the schema, enforced later). No threat surface. (Security-review correctly
deferred to specs 002+.)

**4d Performance** — N/A (no runtime path). Codegen + gates run in seconds. ✓

**4e Testing / E2** — the acceptance mechanism (schema fixtures, codegen no-diff, gate-behavior
scratch-branch checks) is appropriate. **One gap inherited from the checklist (CHK006):** the
FFI-isolation check covers only `core` + `consumer` by name; a future FFI-free crate would silently
escape it. *Suggestion: make the check's covered-crate list an explicit, reviewable artifact.*

**4f Operational readiness** — CI gates are the operational surface and are defined. ✓

**4g Dependencies / E3** — the codegen tools are pinned with versions (verified current). **One
unvalidated assumption (CHK026):** the spec assumes maturin + napi-rs CLI pre-exist from the
bootstrap; moon + Cargo are confirmed, those two are not. *Suggestion: verify availability in
T009/T010 rather than assume.*

## Cross-Lens Insights

**X1 (Product simplification == engineering risk reduction) — codegen scope.** Deferring TS + Rust
*generation* to specs 004/005 (when their bindings/consumers actually exist) would: reduce 001 scope;
avoid generating shapes nothing validates (a wrong-but-uncaught-until-004 risk); and shrink the
determinism surface 001 must stabilize from three toolchains to one. **Counter-argument (why as-is is
defensible):** generating all three now is the most faithful demonstration of C-07 ("one schema, N
languages") and C-01 (structural parity), gives the freshness gate real cross-language coverage from
day one, and de-risks the *pipeline* design before three binding specs depend on it. This is a
genuine judgment call, not a defect — hence a Recommendation, not a blocker.
