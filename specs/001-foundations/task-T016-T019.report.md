  Loads `prompt-definition.schema.json` and calls
  `jsonschema.Draft202012Validator.check_schema(...)`. Exits 0 on success,
  1 on any failure (parse error or schema violation).

  New moon project config (`language: unknown`; NOT a Rust crate).
  Declares `check-schema` and `validate-fixtures` tasks.

### T017 — Accept fixtures (3 files)

- `schemas/jsonschema/fixtures/valid/single-body.json`
  Prompt with name/role/body only, no variants. Exercises default-as-root
  (FR-011). Includes `variables` with `format` constraint, `output_model`,
  and `metadata`.

- `schemas/jsonschema/fixtures/valid/multi-variant.json`
  Prompt with two named variants (`concise`, `structured`), shared `variables`
  with all three provenance values and type constraints, `output_model`, and
  `metadata`. Neither variant is named `default`.

- `schemas/jsonschema/fixtures/valid/variant-with-meta.json`
  Prompt with two variants both carrying opaque `meta` objects (weight/group/
  tags). Root also carries `meta` (default arm selection metadata). Exercises
  `external` provenance on a variable.

### T018 — Reject fixtures (7 files)

- `schemas/jsonschema/fixtures/invalid/bad-role.json`
  `role: "developer"` — not in `[system, user, assistant]` enum.

- `schemas/jsonschema/fixtures/invalid/bad-provenance.json`
  Variable with `provenance: "user-provided"` — not in
  `[trusted, untrusted, external]` enum.

- `schemas/jsonschema/fixtures/invalid/variant-named-default.json`
  `variants` map contains key `"default"` — rejected by `propertyNames/not`
  (FR-011b).

- `schemas/jsonschema/fixtures/invalid/variant-redefines-role.json`
  A variant carries a `role` key — rejected by Variant's
  `additionalProperties: false` (FR-011a).

- `schemas/jsonschema/fixtures/invalid/extra-root-key.json`
  Valid otherwise but contains `unknown_top_level_field` — rejected by root
  `additionalProperties: false`.

- `schemas/jsonschema/fixtures/invalid/missing-required.json`
  Has `name` and `role` but no `body` — rejected as missing required field.

- `schemas/jsonschema/fixtures/invalid/not-json.txt`
  Not valid JSON (YAML-ish text). The validator treats a JSON parse error as
  a valid rejection (distinct from schema-invalid). The script prints
  `parse-error: ...` to distinguish it from a schema-level rejection.

### T019 — Fixture validation script

- `schemas/jsonschema/scripts/validate_fixtures.py`
  - Iterates `fixtures/valid/` → each must pass schema validation.
  - Iterates `fixtures/invalid/` → each must fail (parse error OR schema
    validation error).
  - Prints per-fixture `[PASS]`/`[FAIL]` with a diagnostic note
    (accepted / schema-invalid at [...]: message / parse-error: ...).
  - Prints summary line with total counts.
  - Exits 0 only when all expectations are met; exits 1 otherwise.
  - Wired as moon task `validate-fixtures` with dep on `check-schema`.
