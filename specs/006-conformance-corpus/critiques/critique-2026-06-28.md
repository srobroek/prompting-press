## Engineering Lens Findings

**4a Architecture soundness** — Shared-corpus + 3-runners is the right shape; the moon-task + `scripts/ci`
+ `ci.yml` pattern matches the repo. The found CI gap (the consumer's `cargo test` runs nowhere today) is
correctly turned into a win via the Rust leg (FR-015). The Rust passthrough-`RawVars` newtype is sound and
honestly bounded as zero-engine-logic. **But see E2** on what the Rust leg actually proves.

**4b Failure-mode analysis** — The big one:

> **E1 (Must-Address) — `int-vs-float` as `1 vs 1.0` is unsatisfiable in TypeScript.** JS has one numeric
> type (IEEE-754 double); `1.0` and `1` are indistinguishable (`JSON.stringify(1.0) === "1"`,
> `Number.isInteger(1.0) === true`), and the napi `serde-json` bridge reads an integral JS number as an
> i64 — so a TS-constructed "float 1.0" marshals to the kernel as integer `1` and renders `"1"`, while
> Python's `float(1.0)` marshals as f64 and renders `"1.0"`. The two bindings therefore **cannot** agree
> on `1.0`, and no fixture can make them. As literally exemplified in the spec (`integer vs float (e.g. 1
> vs 1.0)`, FR-002/SC-002) and the plan, the case is mis-specified. This is not a library bug the corpus
> should "catch" (it's an inherent property of JS's number model, not a marshaling defect) — it's a
> fixture-design error that would either wedge the gate red or be unconstructable.

**4c Security/privacy** — N/A surface (test fixtures, non-secret by construction). The SEC-004 posture is
correctly carried into the failure-output requirement (FR-014). No finding.

**4d Performance/scalability** — Small fixture set, synchronous, one Linux leg each. No concern.

**4e Testing strategy** — The runners ARE the tests; cross-check-via-golden is a clean model. Two honesty
gaps:

> **E2 (Recommendation) — the Rust runner's marshaling assertion is near-tautological.** Goldens are
> generated *from the Rust reference binding* (research D3). So the Rust marshaling runner asserts the Rust
> output equals a golden that IS the Rust output — it can only ever fail on nondeterminism, never on a
> parity divergence. The genuine independent parity votes are **Python-vs-golden** and **TS-vs-golden**.
> The Rust leg's real value is (a) the golden *source*, (b) schema round-trip, (c) a determinism guard,
> and (d) closing the CI gap. Recommend stating this explicitly so a reader doesn't over-credit "3-way
> marshaling parity" when it's really "2 independent votes against a Rust-anchored reference + a
> determinism check." (Alternatively, generate goldens from a neutral source — but Rust-anchored is fine
> if labeled.)

> **E3 (Recommendation) — schema-reject parity doesn't assert the rejection *reason* agrees.** FR-009/010
> require the same accept/reject *verdict* and a structured rejection, but not that the normalized `code`
> (or finding kind) matches across bindings for the same invalid doc. Since all three share the closed
> code vocabulary, asserting code-agreement on rejects is cheap and meaningfully strengthens "round-trip
> parity" (it would catch a binding that rejects for the wrong reason). Optional for v1; worth a line.

**4f Operational readiness** —

> **E4 (Recommendation, minor) — T015 lists the `#[ignore]`d goldens test in the gate command.** The gate
> runs `cargo test ... --test conformance_goldens`, but that test is `#[ignore]`d so it's compiled and
> skipped — harmless but confusing (reads as if CI regenerates goldens). Drop `--test conformance_goldens`
> from the gate invocation, or add a one-line comment that it's compile-only and never run in CI.

> **E5 (Question) — golden-regen discipline has no machine guard.** The quickstart correctly warns "don't
> regenerate to make red green," but nothing enforces it; a careless regen silently launders a real
> divergence. Accepted-for-v1 is reasonable (a goldens-freshness gate was deferred in D3), but confirm
> that PR review of the goldens diff is the intended control, and consider a CODEOWNERS/comment nudge on
> `conformance/marshaling/*.json`.

**4g Dependencies/integration** — No new shipped-library dep (correctly forbids a JS decimal lib / second
YAML parser); test-harness helpers pinned exact. Depends only on merged specs 004/005. No finding.

## Cross-Lens Insights

> **X1 (Must-Address, = E1) — the `int-vs-float` case sits at the Product×Engineering seam.** Product: the
> spec's own example (`1 vs 1.0`) is misleading and an implementer will author a fixture that can't pass.
> Engineering: it's structurally unconstructable in JS. The fix is small and the spec already contains the
> mechanism: **redefine the int-vs-float case to use values representable as distinct in all three
> languages** — e.g. assert an integer `1` stays integer-rendered (`"1"`) and a *fractional* float `2.5`
> stays float-rendered (`"2.5"`) — and **explicitly record `1.0`-as-distinct-from-`1` as a known
> JS-unrepresentable boundary** under the spec's existing feature-gap escape hatch (Edge Cases). That
> converts a latent gate-wedge into a documented, intentional scope line.
