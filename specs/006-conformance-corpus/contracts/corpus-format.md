# Contract — Conformance corpus format & runner obligations

This is the contract the shared corpus and the three runners agree on. It is the corpus's "interface":
the fixture-file format, what every runner MUST do, and what the CI gate guarantees. (No public library
API changes — this is a test-artifact contract.)

---

## 1. Corpus layout (single source of truth)

```text
conformance/
├── README.md
├── marshaling/<case>.json        # one marshaling-parity fixture per file
└── schema/
    ├── manifest.json             # references existing schema fixtures + verdicts
    └── yaml/                      # minimal YAML twins (1 valid, 1 invalid) for the load_yaml path
```

There is exactly ONE copy of each fixture. Runners read it; none owns a private copy (clarified Q1).

---

## 2. Marshaling fixture contract

A marshaling fixture conforms to the schema in [data-model.md](../data-model.md#entity-marshaling-fixture):
`{ case, description?, definition, variant?, input, expected }`, where `input` maps each Vars field to a
`{ type, value }` typed descriptor and `expected` is `{ text, template_hash, render_hash }`.

**Required coverage** (SC-002 — at least one fixture per hard case):
`date` (datetime), `decimal`, `nested-model`, `null-undefined-none`, `int-vs-float`.

---

## 3. Runner obligations (identical across Rust / Python / TypeScript)

Every runner MUST, for each marshaling fixture:

1. Load the prompt `definition` into the binding's `Registry` (via `insert` of the constructed object, or
   `load_json` of the serialized definition).
2. Construct the Vars value by mapping each `input` field's `type` to the **native** language value
   (per the D2 table) carrying `value`. `absent` ⇒ omit the field; `null` ⇒ the language's null.
3. Call the binding's **real public render** entry (`render(...)`) with the constructed Vars and `variant`.
4. Assert the result's rendered text equals `expected.text`.
5. Assert the result's `template_hash` / `render_hash` (`templateHash`/`renderHash` in TS) equal
   `expected.template_hash` / `expected.render_hash`.
6. On mismatch, fail with a message naming the **binding, the `case`, and the divergence kind** (text /
   template_hash / render_hash) — FR-014. Failure text may include fixture content (non-secret by
   construction) but MUST NOT add raw bound-value content beyond the fixture (SEC-004 posture).

Every runner MUST, for each schema round-trip fixture in `manifest.json`:

1. Read the document at `path` — resolving it **within the repo/corpus root**; a runner MUST reject an
   absolute path or one that escapes the root via `..` (SEC-001, test-harness path hygiene; the manifest
   is repo-committed, so this is defense-in-depth, not an exploitable boundary).
2. Feed it through the binding's loader matching `form` (`load_json`/`loadJson` or `load_yaml`/`loadYaml`).
3. If `verdict == "accept"`: assert it loads with no error.
4. If `verdict == "reject"`: assert the binding raises its **normalized structured error** — assert on the
   error **type / normalized `code`** (the structured contract), NOT on free-text detail, and never assert
   that scrubbed `parse`/`render`/`excluded_feature` detail is *present* (SEC-002: keep the test from
   coupling to — or pressuring a reviewer to weaken — the SEC-004 scrub). No partial load, no crash
   (FR-010).
5. On a verdict mismatch, fail naming the binding and the fixture (FR-014).

**Forbidden in a runner** (C-02 / Principle I): no rendering, hashing, agreement analysis, or variant
resolution implemented in the runner. The runner only constructs native values, calls the binding's public
API, and compares. (The Rust runner's `RawVars` newtype is Serialize-delegation + a no-op garde Validate —
no engine logic; D4.)

---

## 4. Golden provenance contract (D3)

- `expected.{text, template_hash, render_hash}` are generated by a committed script that renders each
  fixture through the **Rust reference binding** and writes the goldens into the fixtures.
- Runners NEVER regenerate goldens at test time — they only assert equality. This makes cross-binding
  parity transitive (all three == the one golden) and makes the golden a regression tripwire for a
  lockstep kernel change (clarified Q2).
- The golden set stays bounded to the named hard cases; it MUST NOT grow into a comprehensive
  render-parity set (FR-016 / C-01).
- **Honest accounting of the votes (critique E2)**: since the golden is the Rust binding's own output, the
  independent *marshaling*-parity votes are **Python-vs-golden** and **TS-vs-golden**; the Rust marshaling
  assertion is a determinism guard, not an independent parity vote. The Rust leg IS an independent check
  for the **schema round-trip** (the loader is not the golden source). Do not describe the marshaling
  result as "three independent votes."

---

## 5. CI gate contract

- A `ci:conformance` moon task runs all three runners (the Rust leg via `cargo test -p prompting-press
  --test conformance_marshaling --test conformance_schema` — the two gated Rust test targets; the
  `#[ignore]`d `conformance_goldens` regeneration target is NOT gated; the Python/TS runners via their
  existing extension/addon build paths) — D5.
- It is locally reproducible: `mise exec -- moon run ci:conformance` reproduces the CI result (FR-013).
- It is invoked from `.github/workflows/ci.yml` and a marshaling-parity or schema-round-trip divergence
  fails the build (FR-012 / SC-007).
- The Python and TS runners additionally run under the existing `ci:test-python` / `ci:test-node` gates
  (they live in those test dirs); the dedicated task guarantees the **Rust consumer** participates
  (FR-015) — closing the gap that `cargo test -p prompting-press` runs nowhere in CI today.

---

## 6. What this contract does NOT cover (scope guard)

- **Render parity in general** — structural via the single shared core (Principle I / C-01); NOT tested
  here. The spec-002 render fixtures (`crates/prompting-press-core/tests/fixtures/render/`) remain the
  only render-fixture set, byte-unchanged (FR-016).
- **New library capability / public API** — none. The corpus is fixtures + runners + a gate (FR-018).
- **Validation semantics** — the corpus tests marshaling + schema acceptance, not Pydantic/Zod/garde
  validation behavior (that is each binding's own test suite).
