## Engineering Lens Findings

### E1 тАФ ЁЯОптЖТЁЯТб Must-note (downgraded to Recommendation on analysis) тАФ the "three sets" gap
There are **three** variable sets that ought to agree but only two are cross-checked:
1. the template's referenced roots (kernel `required_roots`),
2. the prompt definition's `variables` block (the agreement-check authority тАФ clarify Q1),
3. the caller's garde Vars struct fields (validated, then serialized to the kernel value).

`check()` verifies **(1) тКЖ (2)**. garde verifies the *values* of (3). **Nothing checks (3) agrees with
(2)/(1).** So a Vars struct with a misnamed field (`usrname` vs `username`) validates fine, serializes
to a `Value` with the wrong key, and the kernel renders with the real var **undefined**.

**Why it's a Recommendation, not a blocker**: under the kernel's strict-undefined (FR-001a, spec 002),
that mismatch is a **loud render error**, not a silent empty render тАФ the library's core thesis holds.
The gap is *early detection / DX*: it's caught at render, not by `check()` in CI.

**Suggestion**: (a) document the invariant explicitly тАФ "the caller's Vars struct field names must
match the prompt's declared `variables`; `check()` validates the templateтЖФdefinition agreement, and a
struct/definition mismatch surfaces as a strict-undefined render error" тАФ in the crate rustdoc (T025)
and spec Assumptions; (b) add a test (US1 or a new case) asserting a misnamed-struct-field render
produces a normalized `UndefinedVariable`-class `ConsumerError`, so the behavior is pinned. Optionally
note a future additive `check()` extension that compares a registered struct schema to `variables`
(deferred тАФ would need the type registration clarify Q3 deliberately rejected, so NOT for v1).

### E2 тАФ ЁЯТб Recommendation тАФ synthesized error `code` honesty
garde 0.23 `Error` exposes only `message()` (no machine code); the plan synthesizes `code`
(e.g. `"validation"`). Sound, but make the synthesized codes a small documented closed set (e.g.
`validation`, `unknown_prompt`, `unknown_variant`, `undefined_variable`, `parse`, `render`,
`excluded_feature`, `load`) so consumers can match on `code` stably. Pin the set in `error.rs` + the
contract. (research D3 already notes the synthesis; this just asks for a fixed vocabulary.)

### E3 тАФ ЁЯТб Recommendation тАФ SC-003 YAML/JSON parity is a real assertion, confirm the test does the work
SC-003 demands YAML-loaded and JSON-loaded defs be *identical representations*. serde deserializes both
into the same `PromptDefinition`, so for equivalent docs they SHOULD be equal тАФ but YAML's richer scalar
typing (and the Norway-safe yaml-rust2 behavior) means the test must use genuinely equivalent docs and
assert structural equality (e.g. derive/compare `PartialEq`, or compare re-serialized JSON). T012 V2.2
covers it тАФ just ensure the assertion is field-equality of the parsed structs, not a loose smoke check.

### E4 тАФ ЁЯдФ Question / reviewed-sound тАФ garde feature set
The plan uses `garde = { features = ["derive", "serde"] }`, dropping defaults (email/url/credit_card).
Custom validators (`#[garde(custom)]`) and the standard rules (`length`/`range`/etc.) are **core**, not
behind those features тАФ so dropping them loses nothing the spec needs and trims the dep tree (regex,
idna, url). Confirm at impl time that no test Vars struct uses an `email`/`url`/`credit_card` rule
(they'd need the feature re-added). Reviewed тАФ sound as planned.

### E5 тАФ reviewed-sound тАФ no validation bypass; check() is pure-data
Confirmed (matches the security review): `render`/`resolve` validate-before-render; `check()`/
`get_source` take no Vars, so no value-bearing path skips `validate()`. And `check()` uses
`def.variables` (not the garde struct), so it's genuinely CI-runnable without the caller's Rust types тАФ
the clarify Q1 decision pays off exactly as intended. No finding.

## Product Lens Findings

### P1 тАФ ЁЯТб Reviewed, sound тАФ scope discipline + MVP
No I/O, LLM, request-body, output-parse, built-in token counter, or new pluggable seam (only the
constitution-sanctioned `count_tokens` hook). US1 (validate+render) is a genuine demoable MVP. The
consumer surface (registry/render/check/compose + hook) is complete for a v1 Rust consumer; nothing
obviously missing, nothing in scope the constitution forbids. Recorded sound.

### P2 тАФ ЁЯдФ Question тАФ is `check()`'s CI ergonomics documented for the "user"?
The headline value is `check(registry)` as a CI lint. The spec says it's CI-runnable; consider a
one-paragraph quickstart/rustdoc example showing the CI usage (load registry from repo YAML, run
`check`, non-zero exit on findings) so the differentiator is discoverable. Minor DX. (T025/quickstart.)

## Cross-Lens Insights

### X1 тАФ ЁЯТб (converges with security SEC-004) тАФ error-detail discipline
The normalizer is the one place bound-value content could leak (kernel `Parse`/`Render` detail) AND the
place a misnamed field (E1) surfaces. One consolidated treatment: the error-normalization layer (T005)
should (a) scrub raw detail (SEC-004/FR-015), and (b) produce a clear `undefined_variable` code for the
E1 mismatch case тАФ both are "make the normalized error trustworthy + actionable." Fold into T005 + its
test.
