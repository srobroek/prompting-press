## The validate → serialize → kernel chain

`render` body, in order (every step delegates; the consumer adds no render/agreement/variant/
hash logic — FR-011):

1. `reg.get(name).ok_or_else(|| ConsumerError::UnknownPrompt(name.to_string()))?` — registry
   miss is a structured error, returned **before** validation, never a panic (FR-008a).
2. `vars.validate().map_err(ConsumerError::from)?` — one whole-struct garde pass **before any
   templating** (FR-002). On failure the existing `From<garde::Report>` impl normalizes to
   `ConsumerError::Validation(rows)` and the kernel is never reached.
3. `let values = minijinja::Value::from_serialize(vars);` — bridge the now-validated struct to
   the kernel value type (FR-003a). `from_serialize` takes `T: Serialize` by value; we pass
   `vars: &V`, and `&V: Serialize` holds, so no clone. (Matches kernel-side usage of
   `Value::from_serialize` verified in `prompting-press-core` tests.)
4. `prompting_press_core::render(def, variant, values, guard).map_err(ConsumerError::from)` —
   delegate; the existing `From<KernelError>` impl normalizes any kernel failure.

`get_source` has no vars: resolve name → `prompting_press_core::get_source(def, variant)
.map_err(ConsumerError::from)`. No validation, no bridging.

Validation-fails-before-render is structurally guaranteed: step 2 is a `?` early-return, so a
`Validation` error is returned *before* `from_serialize`/the kernel call in steps 3–4 ever run.
The test asserts this by matching on the `Validation` variant (a `Kernel` variant would prove a
render was attempted).

## Three-sets gap (E1) — how it surfaces

A Vars struct field name that disagrees with the prompt's declared `variables`
(struct `usrname` vs template/`variables` `username`):

- garde `validate()` **passes** — the value (`"ada"`, non-empty) is valid; garde validates
  *values*, not name agreement. The test asserts `vars.validate().is_ok()` explicitly.
- `Value::from_serialize` serializes the struct, so the kernel value map carries `usrname`,
  **not** `username`. The template's `{{ username }}` is therefore undefined.
- The kernel renders under `UndefinedBehavior::Strict`, so it raises `KernelError::Undefined
  Variable`, which the consumer's `From<KernelError>` maps to a `FieldError` with
  `code = "undefined_variable"`.

Result: `Err(ConsumerError::Kernel(rows))` where some row has `code == code::UNDEFINED_VARIABLE`
— a **loud, normalized** error, never a silent empty render. This pins the documented invariant
(spec Assumptions / critique E1) by test rather than by an extra in-library check (closing it
in-library would need the per-prompt type registration clarify Q3 rejected for v1).

## Guard plumbing (F5)

The consumer plumbs `guard: &GuardConfig` straight through to the kernel and surfaces the
kernel's `RenderResult.guard: Option<String>` unchanged. The test asserts only plumbing:
`GuardConfig::default()` (disabled) → `guard.is_none()`; `GuardConfig { enabled: true, .. }`
over a prompt declaring an `untrusted` field → `guard.is_some()`; and `text` is identical
either way (the guard is additive). Guard *wording/expansion* is the kernel's contract
(spec 002) and is not re-tested here. Note: the enabled-guard test prompt declares `name`
as `untrusted`, because the kernel's `build_guard_text` returns `None` over an empty
untrusted∪external union — without a tagged field, `enabled: true` would still yield `None`.
