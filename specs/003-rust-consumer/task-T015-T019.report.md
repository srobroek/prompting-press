## The types (T017)

```rust
pub struct CheckReport { pub findings: Vec<Finding> }      // empty ⇒ pass
impl CheckReport { pub fn passed(&self)->bool; pub fn is_empty(&self)->bool; }

pub struct Finding {
    pub prompt: String,
    pub variant: Option<String>,   // Some("default")/Some("<name>") for agreement/analysis; None for provenance
    pub kind: FindingKind,
    pub detail: String,            // actionable (FR-020), no bound-value content

pub enum FindingKind {
    UndeclaredVariable { name: String },     // agreement (FR-016/017)
    UntrustedWithoutGuard { field: String }, // provenance (FR-018, reframed)
    AnalysisError { reason: String },        // a required_roots() Err (see below)
}

pub fn check(reg: &Registry) -> CheckReport;  // pure, infallible

## Agreement lint (T018, FR-016/017)

`variants_to_check(def)` yields `["default", <named variants sorted>]` — `def.variants` is a
`HashMap`, so the named keys are collected through a `BTreeSet` to make the report deterministic.
For each, `"default"` maps to `required_roots(def, None)`, a named variant to
`required_roots(def, Some(name))`.

The kernel computes the referenced roots; the consumer owns **only the subtraction**
(constitution Principle I / C-01 / FR-017): `agreement.required_roots` (a sorted `BTreeSet<String>`)
minus the declared set `def.variables.keys()`. Each leftover root → one
`FindingKind::UndeclaredVariable { name }` carrying `prompt` + `Some(variant)`. Leftovers are
emitted in sorted order because `required_roots` is a `BTreeSet`.

## Provenance lint (T019, FR-018 — REFRAMED) and the `meta.guard` convention

The spec-002 kernel has **no in-template guard-position concept**, so the literal "untrusted field
used outside a guard position" is not implementable. This crate adopts the documented,
implementable interpretation (module docs in `check.rs`):

> A prompt has a guard configured **iff a top-level `"guard"` key is present in its `meta` map OR
> its `metadata` map.**

Mechanics:

- `declared_untrusted = provenance_view(def).untrusted ∪ provenance_view(def).external`
  (both sorted `BTreeSet`s; chained into a fresh sorted, de-duplicated set).
- If `declared_untrusted` is empty → no obligation, return.
- If `has_guard_configured(def)` (`def.meta.contains_key("guard") || def.metadata.contains_key("guard")`)
  → satisfied, no finding.
- Otherwise emit one `FindingKind::UntrustedWithoutGuard { field }` per uncovered field, with
  `variant: None` (the obligation is prompt-level, not per-arm).

`meta`/`metadata` are library-opaque `serde_json::Map`s; the lint reads them **read-only** and
checks only for the *presence* of the `"guard"` key — it never interprets the guard's contents (that
is the caller's concern; render-time `GuardConfig` is what actually drives guard expansion in the
kernel). This is pure analysis: `provenance_view` is pure and the maps are only read.

## How a `required_roots` Err is handled (choice noted)

`check()` must stay infallible (`-> CheckReport`, not `Result`) so it is a drop-in CI gate (F7). A
`required_roots` `Err` (malformed template, or an excluded feature such as `{% include %}`/macros/
inheritance, which never parse under the kernel's feature set) is therefore **recorded as a finding**
with a distinct third kind `FindingKind::AnalysisError { reason }` rather than propagated. This keeps
`check()` infallible while still **failing the gate** on an un-analyzable template (it does not get
silently swallowed). `reason` is a stable, **scrubbed** code derived from the `KernelError` variant
class (`"parse"`, `"excluded_feature"`, `"unknown_variant"`, ...) — the kernel's raw `detail` (which
may carry bound-value content, SEC-004/FR-015) is never copied into the finding.

This is the "distinct kind" option the task explicitly permitted; the two C-04/C-09 lint classes
named in T017 (`UndeclaredVariable`, `UntrustedWithoutGuard`) are unchanged, `AnalysisError` is
additive.

## Determinism

Registry iterates by name (`BTreeMap`); variants visited default-first then named-sorted (`BTreeSet`);
undeclared roots sorted (`BTreeSet` from kernel); provenance fields sorted (kernel `ProvenanceView`
sets). Findings order is reproducible.
