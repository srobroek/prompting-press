## Engineering Lens Findings

### E1 — 🎯 Must-Address — MiniJinja feature set accidentally drops `adjacent_loop_items` (Dependencies)
`research.md` D1 / `plan.md` Technical Context / task T001 specify
`minijinja = { version = "2.21", default-features = false, features = ["builtins", "deserialization",
"serde", "std_collections"] }`. Relative to 2.21.0's default set, this intentionally drops `macros` +
`multi_template` (the FR-002 mechanism — correct) but **also silently drops the default-on
`adjacent_loop_items`** and `debug`.

**Verified against 2.21.0 source** (`minijinja/src/vm/loop_object.rs`): `adjacent_loop_items` gates
**only** `loop.previtem`, `loop.nextitem`, and `loop.changed()`. The core loop accessors
(`loop.index`, `index0`, `first`, `last`, `length`, `revindex`) are NOT gated and work without it. So
basic loops are **not** broken — but a v1 template using `loop.previtem`/`nextitem`/`changed` would
fail, and the spec's "loops" feature scope (FR-001) doesn't say those are excluded. Dropping it looks
accidental, not designed.

**Suggestion**: Re-add `adjacent_loop_items` to the feature set (it is pure-Rust, no FFI, default-on —
keeping it makes "loops" mean full Jinja loops, the unsurprising choice). Update research D1, plan
Technical Context, and T001. Separately decide `debug`: it improves `Error` line/context detail (helps
FR-028 error quality) but also registers a `debug()` global that would then need an allowlist entry —
recommend leaving `debug` OFF and noting it, since FR-020's env-derived allowlist handles globals
either way.

### E2 — 💡 Recommendation — Elevate the parse-error short-circuit to an FR (Soundness)
`Template::undeclared_variables` returns an **empty set on parse failure** (verified in 2.21.0 source:
`Err(_) => HashSet::new()`). research D2 and task T025 already require the agreement analysis to
parse-and-short-circuit to a `Parse`/`ExcludedFeature` error *before* trusting the undeclared set — so
the design is not actually exposed. But this is the soundness of the **headline feature**; encoding it
only in a task + research note is thin.

**Suggestion**: Add a one-line FR (e.g. FR-016a) making "the analysis MUST error on a non-parseable
template, never return an empty required-roots set" a normative requirement, with T025 as its
regression guard. Cheap rigor on the load-bearing guarantee.

### E3 — 💡 Recommendation (was flagged Must-Address; downgraded on verification) — Relocation blast radius (Build/CI)
The generated-shape relocation (T003–T009) is the highest-risk change. The inferred critique feared an
unmapped input would leave a binding/consumer CI job red. **Verified**: the binding crates
(`prompting-press-py`, `prompting-press-node`) contain **zero** references to the generated shape
(`rg` clean), and the consumer re-export + codegen script + moon task + freshness-gate inputs are each
already mapped to a specific task (T004/T005/T007/T008). Blast radius is smaller than feared and is
covered.

**Suggestion**: Keep T009 as the end-to-end verification gate (build workspace + freshness gate green)
and ensure the relocation lands as one atomic commit so `main`/the branch is never left non-building
mid-move. No spec change needed.

### E4 — 💡 Recommendation — Determinism: confirm `BTreeSet` at every exposed boundary (Failure modes)
data-model.md correctly uses `BTreeSet` for `required_roots`, `untrusted`, `external` (sorted →
deterministic). Worth an explicit cross-check that no `HashSet` leaks into any *exposed* output
(internal use is fine). Determinism is the C-01/SC-001 contract.
**Suggestion**: Add a note in T023/T028 to assert sorted output in tests; quickstart V2.1 already
implies it.

### E5 — 🤔 Question — `ExcludedFeature` vs `Parse` error-kind distinguishability (Correctness)
research D4 flags that if MiniJinja's disabled-feature parse error isn't distinguishable by
`ErrorKind`, the kernel falls back to labelling it `Parse`. That's loud and correct (SC-008 still
holds — the template is rejected), just less specific.
**Suggestion**: Acceptable as-is. During T032, capture the actual `ErrorKind` for a disabled-feature
tag and label `ExcludedFeature` if possible; document the fallback if not. No blocker.

## Product Lens Findings

### P1 — 💡 Reviewed, sound — Scope discipline holds
No I/O, LLM, request-body, token-count, output-parse, telemetry, or new pluggable seam is introduced.
US1 (render + provenance) is a real, demoable MVP for the "user" = downstream consumer/binding author.
The narrow boundary is correctly drawn; nothing should be added. (Recorded as sound, not a finding.)

### P2 — 🤔 Question — Is `get_source` enough of a consumer surface?
The kernel exposes `render`, `get_source`, `required_roots`, `provenance_view`. For a kernel feeding
specs 003–007 this looks complete, but confirm the consumer (003) won't need, e.g., a way to enumerate
variant names or read `role`/`output_model` through the kernel vs. straight off the `PromptDefinition`.
**Suggestion**: Likely fine — those are plain fields on the (now kernel-owned) generated shape the
consumer can read directly. No change now; revisit at spec 003.

## Cross-Lens Insights

### X1 — 💡 (converges with security SEC-002) — "surface-but-don't-enforce" footgun
Three kernel outputs *surface* without *enforcing*: provenance tags (metadata, no runtime guard),
the guard field (additive, no sanitization), and `output_model` (reference, never parsed). A consumer
could infer enforcement. The security review raised the same (SEC-002, MODERATE).
**Suggestion**: One consolidated rustdoc section (folds into T033) — "what the kernel does NOT do with
what it returns" — stating guard does not sanitize, tags are not enforced, output_model is not parsed.
Carry into consumer specs.
