### Variant resolution rule (T016, FR-007..FR-011)

- `None` or `Some("default")` → `name = "default"`, `source = &def.body` (root body is
  ALWAYS the default; for single- and multi-variant prompts alike). No "missing default"
  error path exists (FR-010 amended).
- `Some(k)` with `k` a declared variant → `name = k`, `source = &variants[k].body`.
- `Some(k)` with `k` neither `"default"` nor declared → `Err(UnknownVariant { requested: k })`
  — the only variant-resolution error (FR-009).

`get_source` (T017) reuses `resolve_variant` and returns its borrowed `source` — the exact
bytes `template_hash` is computed over.

## Error mapping (strict-undefined + unknown-variant + excluded-feature)

`render` builds a fresh `build_environment()` (strict undefined, `macros`/`multi_template`
off), eagerly parses the resolved source via `add_template_owned`, then renders. Every
`minijinja::Error` goes through `map_minijinja_error`, keyed on `ErrorKind` (verified
against minijinja 2.21.0 source):

- **Unknown variant** — never reaches minijinja; returned by `resolve_variant` before any
  environment is built → `KernelError::UnknownVariant { requested }`.
- **Strict undefined** (`ErrorKind::UndefinedError`) → `KernelError::UndefinedVariable { name }`.
  Verified in 2.21.0 source (`vm/mod.rs` `Emit`, `utils.rs`): the undefined error is built
  as `Error::from(ErrorKind::UndefinedError)` with **no variable-name payload**. So `name`
  is best-effort: `err.detail()` if `Some`, else `err.to_string()` (still informative). The
  V1.7 fixture asserts the *variant* (`matches!(.., UndefinedVariable { .. })`), not the
  exact name text, per the task note — so this is exact to spec.
- **Excluded feature / parse** (`ErrorKind::SyntaxError`) → `ExcludedFeature { detail }` when
  the message names an excluded keyword (`include`/`import`/`from`/`extends`/`macro`/`block`,
  quoted), else `Parse { detail }`. Heuristic only refines the label; both are loud
  parse-time errors, so a miss is benign (research D4). (Exercised by US-edge tasks T013+
  excluded-feature fixtures later; the mapping is in place now.)
- **Any other render failure** → `Render { detail }`.

## Hashing approach (T018, FR-012/013/014, research D8)

`hashing::sha256_hex(s)` = `format!("{:x}", Sha256::digest(s.as_bytes()))` (lowercase hex,
64 chars). In `render`: `template_hash = sha256_hex(resolved.source)` (the unrendered
variant source string, identical to what `get_source` returns) and
`render_hash = sha256_hex(&text)` (the rendered output). No `vars_hash`, no hash over
structured input (FR-014). Unit tests pin the NIST vectors for `""` and `"abc"`.

The `tests/hashing.rs` suite re-computes SHA-256 independently (its own `sha256_hex` over
`sha2`) and asserts `template_hash == sha256(get_source(..))` and `render_hash == sha256(text)`
(V1.8), proving the kernel hashes the right strings rather than trusting its own helper.
