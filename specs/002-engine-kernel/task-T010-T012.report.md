## T010 — `KernelError` enum (verbatim)

```rust
#[derive(Debug)]
#[non_exhaustive]
pub enum KernelError {
    UnknownVariant { requested: String },     // FR-009
    ExcludedFeature { detail: String },        // FR-002 (include/import/extends/macro/inheritance)
    Parse { detail: String },                  // template syntax parse failure
    UndefinedVariable { name: String },        // FR-001a (strict-undefined hit at render)
    Render { detail: String },                 // other render-time failure
}
```

Variants are exactly the five in data-model.md §KernelError, with the field names
specified there. `Display` writes clear human messages (e.g. `unknown variant:
\`concise\``, `undefined variable at render: \`article\``); `std::error::Error` is
implemented via the empty default. The kernel returns this **native** type — no
normalization to `[{field, code, message}]` here (that is the consumer's job, C-06 /
Principle VI). Two unit tests assert the `Display` strings and that the type satisfies
`std::error::Error`.

### Design note — `#[non_exhaustive]`

I marked the enum `#[non_exhaustive]`. This adds no variants beyond the five required;
it is the idiomatic guard for a public library error enum so a future variant is not a
breaking change. The consumer's normalization match must include a wildcard arm. If you
prefer the enum closed (so the consumer gets exhaustiveness checking and a compile error
when a variant is added), say so and I will drop the attribute — it is the one
judgment call in this file that the spec did not pin.

## T011 — `build_environment` body (verbatim)

```rust
#[allow(dead_code)] // wired into render / analysis in later spec-002 tasks (T013+).
pub(crate) fn build_environment() -> minijinja::Environment<'static> {
    let mut env = minijinja::Environment::new();
    env.set_undefined_behavior(minijinja::UndefinedBehavior::Strict);
    env
}
```

- `'static` is the lifetime that compiles cleanly: the env owns no borrowed template
  source at construction (templates are added per-operation against borrowed definition
  bytes in later tasks).
- `set_undefined_behavior(Strict)` per research D3 / FR-001a (verified present in
  minijinja 2.21.0 `environment.rs:520`; the `undefined_behavior()` getter at `:529` is
  used by the unit test). Under Strict, using/printing an undefined var is a loud error;
  `is defined` still works for optional refs.
- A module doc-comment explains that `macros` / `multi_template` being disabled at the
  **crate-feature** level (workspace `Cargo.toml`) is what makes `{% include/import/
  extends/macro/block %}` parse errors (FR-002, research D1/D4) — so no loader/AST-walk
  is needed — and warns not to re-enable those features.
- `#[allow(dead_code)]` because `build_environment` is not yet called by render/analysis;
  a unit test (`builds_with_strict_undefined`) also calls it, asserting the Strict config.

## T012 — fixture harness approach

Two loader entry points in `tests/common/mod.rs`:

1. `load_regression_case(name) -> RegressionCase` — reads
   `tests/fixtures/render/<name>.json` and deserializes `{description, template, values,
   expected}`. This is the `(template, values) -> expected` regression-case path.
2. `load_prompt_definition(name) -> PromptDefinition` — reads
   `schemas/jsonschema/fixtures/valid/<name>.json` (the spec-001 corpus, loaded **in
   place**, not duplicated) and deserializes via `prompting_press_core::PromptDefinition`
   (FR-027 — the kernel consumes the spec-001 shape).

Paths are resolved from `CARGO_MANIFEST_DIR`; the repo root is `ancestors().nth(2)` of
the crate dir (`crates/prompting-press-core` → `crates` → repo root). Deserialization
uses the crate's existing `serde_json` dependency, so **no YAML or other dev-dependency
was added** (the workspace has no `serde_yaml`; JSON keeps the harness dependency-clean).
The module is `#![allow(dead_code)]` because each future suite uses a subset of the API.

Starter data: `interpolation.json` and `conditional-loop.json`. The scaffold test
(`tests/scaffold.rs`) exercises both loaders: one render case loads and has a non-empty
template/expected + object values; `single-body` and `multi-variant` schema fixtures
deserialize into `PromptDefinition` and expose the expected `name`/`variants`.

### Self-referential-grep mitigation (spec-001 lesson)

The risk: a CI forbidden-pattern grep over Rust **source** (for excluded template
features `{% include %}` etc., or interpolation markers) tripping on the test corpus's
own example templates.

Mitigation applied: **every fixture template body lives in a `.json` data file** under
`tests/fixtures/render/`, deserialized at runtime — never inlined as a Rust string
literal. The `.rs` sources (`common/mod.rs`, `scaffold.rs`) carry only loader logic and
path strings; they contain **no** executable/fixture template strings, no `{{ … }}` or
`{% … %}` literals. `tests/fixtures/README.md` records this intent, including how to add
a future excluded-feature-rejection fixture the same way (offending tag in a data file;
test asserts rejection).

Verification grep (scoped to `**/*.rs` in the crate):
- No template tags or interpolation markers appear in any **fixture/executable** Rust
  string. The only `{%`/`{{` matches in `.rs` are **doc-comment prose** in `error.rs`
  and `engine.rs` (naming the excluded tags inside backticks, exactly as data-model.md
  and kernel-api.md do) and the mitigation note in `common/mod.rs`. None is a fixture
  template body.

Current-CI cross-check: there is **no** forbidden-template-feature grep in CI today
(`.github/workflows/ci.yml`, `scripts/ci/*`); the only Rust-relevant gate is
`check-ffi-isolation.sh`, which is a `cargo tree -i` dependency scan and does **not**
pattern-scan `.rs` for template tags. So the doc comments are safe now, and when the
FR-002 grep lands (later task) it should target fixture data / template fields, which is
exactly where this harness keeps every template.
